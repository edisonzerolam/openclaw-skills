## debug 核心原则

> 融合 Karpathy "Think Before Coding" 框架 + 8个专家知识来源

---

### P1 先诊后断

**来源**：Karpathy Think Before Coding + 结构化日志原则

**适用场景**：
- 用户提交 stacktrace / crash log
- 报告"程序崩溃/错误/异常"
- 任何需要根因分析的场景

**具体做法**：
1. 用自然语言描述"预期行为 vs 实际行为"
2. 找出第一个错误发生的时间戳
3. 识别错误模式（OOM/Segfault/Timeout/FATAL）
4. 二分定位：可能是A问题 / 不可能是A问题
5. 逐一验证，排除不可能项

---

### P2 最小化干预

**来源**：DevOps最佳实践 + Karpathy最小化测试用例

**适用场景**：
- 准备修复方案时
- 用户要求"帮我改代码"
- 需要同时修改多个参数

**具体做法**：
1. 先读日志，再跑命令 —— 不要还没看日志就开始重启服务
2. 最小化复现路径 —— 找到触发bug的最简条件再分析
3. 隔离变量逐一排查 —— 不要同时改多个参数
4. 回滚准备 —— 每次修改前知道怎么恢复
5. 验证后才提交 —— 修改后先在小范围验证再推广

---

### P3 分层知识加载

**来源**：BytesAgain L1-L4分层架构 + 触发词匹配机制

**适用场景**：
- SKILL.md首次加载
- 诊断/错误/修复触发L2
- 调试/验证/脚本触发L3
- --expert 或复杂关键词触发L4

**具体做法**：
1. L1（核心）：6命令快速理解 + 基本用法
2. L2（热数据）：健康检查标准 + 12类错误模式库
3. L3（冷数据）：Python验证脚本 + 调试脚本
4. L4（专家）：references/ 按需加载，不提前加载

---

### P4 提示链引导推理

**来源**：DAIR.AI PE Guide + Context Engineering

**适用场景**：
- 复杂错误诊断需要多步推理
- 日志内容超过500行
- 根因不明显需要层层剥开

**具体做法**：
1. 使用 Chain-of-Thought：分步描述推理过程
2. 提供 Few-Shot 示例：3-5个典型错误案例作为模式参考
3. ReAct模式：推理步骤 + 工具调用验证交替进行
4. 上下文压缩：长日志先做预处理（grep/filter），保留关键信息

---

### P5 并行执行加速

**来源**：BytesAgain [[PARALLEL]]机制 + CAMEL多Agent协作

**适用场景**：
- 多进程同时监控（leaks --pid A & --pid B）
- 多命令同时度量（profile --repeat 5 x2）
- 多端点同时探测（http url1 & url2）
- 多模式并行扫描（trace --pattern A & --pattern B）

**具体做法**：
1. 识别任务中的并行机会（无依赖子任务）
2. 使用 `[[PARALLEL]]` 标记通知主会话分发
3. 各子会话独立超时、独立执行
4. 主会话汇总结果，聚合输出
5. 超过5个并行任务时自动收敛（避免资源耗尽）
