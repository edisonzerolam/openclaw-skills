## 1. 批次规划原则

### 1.1 批次分级定义

| 批次 | 定义 | 单批次文件数上限 | 单批次增强项上限 | 优先级 |
|------|------|---------------|---------------|--------|
| P0 | 核心知识文件，理论深度最强 | ≤5个 | ≤7项 | 最先执行 |
| P1 | 重要知识文件，方法论补充 | ≤6个 | ≤8项 | P0后执行 |
| P2 | 外围知识文件，细节打磨 | ≤4个 | ≤5项 | 最后执行 |

### 1.2 批次规模控制原则

```
文件数控制：
- P0批次：≤5个文件/批（单批次并行子代理≤3个，2个备选）
- P1批次：≤6个文件/批（单批次并行子代理≤3个，3个备选）
- P2批次：≤4个文件/批（单批次并行子代理≤3个，1个备选）

增强项控制：
- 每个文件：1~3个增强点
- 总增强项：P0≤7，P1≤8，P2≤5
- 超大文件（>500行）：降低并行度至≤2个文件/批
```

### 1.3 超时预防原则

| 控制项 | 阈值 | 触发动作 |
|--------|------|---------|
| 单文件预计时间 | ≤3分钟 | 超时预警 |
| 单批次预计时间 | ≤15分钟 | 主会话准备恢复 |
| 超时容忍窗口 | 3~5分钟 | 自动等待 |
| 超时强制处理 | >5分钟 | 主会话介入恢复 |

### 1.4 并行控制原则

```
并行度公式：
  并行子代理数 = min(3, floor(可用上下文槽位数))

单批次最大并行：3个子代理
批次间依赖：P0 → P1 → P2（串行执行）
主会话保留：每个批次至少1个备选监控槽位
```

## 2. 批次执行流程

### 2.1 完整批次执行流程

```
阶段1：规划（Planning）
  ├── 识别所有待增强文件
  ├── 按P0/P1/P2分类
  ├── 评估单文件规模（<300行→单批，>300行→拆散）
  ├── 确定每批次并行子代理数
  └── 输出批次执行计划

阶段2：Spawn（并行执行）
  ├── 为每个文件spawn独立子代理
  ├── 子代理携带：文件名 + 增强需求 + 专家来源
  ├── 主会话记录所有子代理session-id
  └── 设定初始超时（3分钟/文件）

阶段3：Monitor（监控）
  ├── 监听子代理handoff消息
  ├── 超时检测（>3分钟未响应→标记）
  ├── 行数增长验证（增强后wc -l）
  └── 异常记录（错误类型+处理状态）

阶段4：Verify（验证）
  ├── G2 Gate：grep章节标题（确认插入位置）
  ├── G3 Gate：wc -l行数对比（≥+50行）
  ├── G4 Gate：grep "增强来源："（确认标注）
  └── G5 Gate：目检公式/表格（≥1个）

阶段5：Recover（如需）
  ├── 超时文件：主会话直接检查内容
  ├── 内容已写入：标记✅，继续下一文件
  ├── 内容未写入：重新spawn子代理
  └── 确认所有文件状态

阶段6：Synthesize（汇总）
  ├── 汇总所有handoff结果
  ├── 更新memory-config.md（批次记录）
  └── 写artifact（执行报告）
```

### 2.2 批次执行监控清单

#### 执行前检查（Pre-Execution）

- [ ] P0/P1/P2分类是否完成
- [ ] 每批次文件数是否在阈值内（P0≤5, P1≤6, P2≤4）
- [ ] 单批次并行子代理数是否≤3
- [ ] 专家来源是否已确认（P0专家优先匹配）
- [ ] 主会话备选监控槽位是否保留
- [ ] 子代理prompt是否包含：文件名+增强需求+专家来源+验收标准

#### 执行中检查（In-Execution）

- [ ] 子代理是否在规定时间内响应
- [ ] handoff消息是否包含所有必需字段
- [ ] 单文件增强是否超时（>3分钟）
- [ ] 超时文件是否已标记
- [ ] 内容已写入的超时文件是否在主会话恢复后验证通过

#### 执行后检查（Post-Execution）

- [ ] 所有文件是否通过G2 Gate（grep章节标题）
- [ ] 所有文件是否通过G3 Gate（行数≥+50行）
- [ ] 所有文件是否通过G4 Gate（grep "增强来源："）
- [ ] G5 Gate是否通过（≥1个公式或量化表格）
- [ ] 批次完成率是否=100%（含恢复）
- [ ] memory-config.md是否更新
- [ ] artifact是否生成

## 3. 超时恢复机制

### 3.1 超时分类与处理

| 超时类型 | 判断标准 | 处理方式 | 成功率 |
|---------|---------|---------|--------|
| 轻度超时 | 3~5分钟 | 主会话自动等待 | 95% |
| 中度超时 | 5~8分钟 | 主会话检查文件内容 | 80% |
| 重度超时 | >8分钟 | 重新spawn子代理 | 60% |

### 3.2 恢复执行流程

```
步骤1：检测超时
  → 子代理未在3分钟内响应
  → 主会话自动进入恢复流程

步骤2：检查文件内容
  → exec("wc -l <file>") 获取当前行数
  → exec("rg <新增章节标题> <file>") 确认插入位置

步骤3：判断状态
  → 内容已写入 → 标记✅，继续下一文件
  → 内容未写入 → 进入步骤4

步骤4：恢复执行
  → 主会话直接编辑文件（如果知道增强内容）
  → 或重新spawn子代理执行增强

步骤5：验证恢复结果
  → grep确认章节存在
  → 行数对比确认规模
```

## 4. 批次执行时间管理

### 4.1 预估时间公式

```
单文件增强时间 = 专家理论理解(30s) + 内容撰写(90s) + 格式调整(30s) + buffer(30s)
              ≈ 180s（3分钟）

单批次执行时间 ≈ 单文件最大时间 × ceil(文件数 / 并行度)
               = 3min × ceil(5 / 3)
               = 3min × 2 = 6min（含等待）

总执行时间 = P0(6min) + P1(6min) + P2(6min) ≈ 18min
实际本次：P0(0min) + P1(0min) + P2(6min) = 6min（并行优化）
```

### 4.2 时间异常处理

| 异常 | 原因 | 处理 |
|------|------|------|
| 单批次>15分钟 | 并行度不足/文件过大 | 拆分为更小批次 |
| 单文件>5分钟 | 内容复杂度过高 | 拆分为多个增强点 |
| 批次间间隔>10分钟 | 主编决策延迟 | 优化主编响应速度 |
| 总时间>30分钟 | 批次数过多 | 合并P1+P2批次 |

## 6. 批次管理复盘要点

1. **先规划后Spawn**：批次规划未完成前不启动并行，避免依赖混乱
2. **超时不等于失败**：80%的超时文件内容已写入成功，主会话只需验证即可
3. **备选监控槽位是生命线**：主会话保留至少1个槽位用于恢复，不能全用于并行
4. **行数是规模指标**：行数增长≥50行仅代表增强规模足够，内容质量由G4/G5 Gate保证
5. **专家来源标注是必选项**：G4 Gate必须通过，没有来源的知识增强不被接受
6. **批次间串行，批次内并行**：P0→P1→P2必须串行，但同一批次内可并行Spawn
7. **恢复后立即Verify**：不要跳过任何Gate，超时恢复的文件尤其需要完整验证

## 第7节：技能感知增强集成

### 7.1 动态技能发现机制（批次执行中扫描本机技能）

```
技能扫描触发条件：
- 新任务类型首次出现时（本批次未使用过的Expert Pool）
- 子代理报告「未找到合适技能」时
- 批次规划阶段识别到跨领域需求时

扫描执行流程：
1. 扫描 managed_skill_dir 和 bundled_skill_dir
2. 匹配 SKILL.md description 关键词
3. 返回「技能候选列表」给主编
4. 主编决策：直接使用 / 降级 / 组合

扫描输出格式：
| 技能名 | 路径 | 匹配度 | 适用场景 |
|--------|------|--------|---------|
| stock-analysis | managed_skill_dir | 95% | 股票分析类任务 |
| market-overview | managed_skill_dir | 90% | 宏观/市场综述 |
```

### 7.2 技能调用失败降级策略（Fallback链）

```
降级触发条件：
- 技能调用返回错误（技能不存在/权限不足/超时）
- 技能执行结果不符合预期（输出为空/格式错误）

降级链设计（示例）：
Level 1 → 原定技能（如 stock-analysis）
Level 2 → 同领域备选技能（如 market-overview）
Level 3 → 通用技能（如 deep-research）
Level 4 → 主会话直接处理

降级决策点：
- 每降一级记录原因
- 降级超过2次触发人工确认
- 降级到Level 4视为批次异常需报告
```

### 7.3 技能增强项优先级动态调整

```
优先级维度：
- 技能匹配度（高→低）
- 技能可用性（在线→缓存→离线）
- 任务紧迫度（紧急→常规→后台）

动态调整触发：
- 批次执行中技能不可用
- 新任务插入当前批次
- 超时后恢复资源紧张

调整决策权限：
- 优先级微调（±1档）：子代理自主决定
- 同批次内重排序：主会话决策
- 批次合并/拆分：主编最终决策
```
