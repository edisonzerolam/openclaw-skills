## 1. 知识工程团队角色定义

| 角色 | 英文 | 职责 | 模型选择 |
|------|------|------|---------|
| 主编 | Orchestrator | 规划增强批次、监督执行、验证结果、写MEMORY | 高推理模型（如 MiniMax-M2.7） |
| 领域专家 | Expert | 提供真实专家理论/公式/数据来源 | 高推理模型 |
| 知识工程师 | Knowledge Engineer | 将专家理论转化为知识文件格式 | 中等成本模型 |
| 验证员 | Verifier | grep验证、行数统计、内容审查 | 低成本模型 |

### 角色协作图

```
主编（Orchestrator）
  ├── 接收任务（Inbox）
  ├── 规划批次（P0/P1/P2）
  ├── 分配给知识工程师
  ├── 接收handoff汇报
  └── 终审 + 写MEMORY

领域专家（Expert）
  └── 提供理论/公式/来源

知识工程师（KE）
  ├── 编辑增强目标文件
  ├── 插入专家来源标注
  └── 报告handoff给主编

验证员（Verifier）
  ├── grep验证章节标题
  ├── 行数增长统计
  └── 内容可信度审查
```

## 2. 知识工程团队任务生命周期

```
Inbox（任务进入）
  │
  ▼
Planning（批次规划：P0/P1/P2分类）
  │  P0：核心文件 ≤5个/批，理论深度最强
  │  P1：重要文件 ≤6个/批，方法论补充
  │  P2：外围文件 ≤4个/批，细节打磨
  │
  ▼
Spawn（并行spawn子代理执行增强）
  │  每批次并行 ≤3个子代理
  │  每个子代理处理 1~2 个文件
  │  主会话保留备选监控
  │
  ▼
Monitor（子代理执行监控：超时检测、进度跟踪）
  │  超时阈值：≈3分钟/文件
  │  超时处理：主会话恢复 or 跳过
  │  进度跟踪：子代理完成后即时handoff
  │
  ▼
Verify（grep验证：章节标题 + 行数增长）
  │  检查1：新增章节标题是否在文件内（grep确认）
  │  检查2：行数增长是否 ≥ 增强前+50行
  │  检查3：是否含专家来源标注（grep "增强来源："）
  │  检查4：是否含 LaTeX 公式或量化表格
  │
  ▼
Synthesize（汇总结果、更新MEMORY、写artifact）
  │
  ▼
Done（完成）
```

## 3. 知识工程的Handoff协议

### Handoff消息格式

当子代理完成时，必须向主编汇报以下字段：

```
✅ 增强完成: <文件名>
- 新增章节: <章节标题>
- 文件行数: <增强后行数>行（增强前<增强前行数>行）
- 专家来源: <专家姓名>（<机构>）<年份>年<领域>方法
- 状态: <已通过grep验证|部分验证|待审查>
- 已知问题: <超时/内容缺失/待补充>（如有）
```

### Handoff示例

```
✅ 增强完成: hk-stock-analysis.md
- 新增章节: 第3.6节「港股创新药管线概率树估值模型」
- 文件行数: 308行（增强前262行）
- 专家来源: 黄咏衫（高盛香港）2023~2024年港股18A估值方法
- 状态: 已通过grep验证（第188行确认）
```

### 异常Handoff

```
⚠ 超时但内容已写入: content-director.md
- 新增章节: 第4.3节「抖音流量池算法推演」
- 文件行数: 447行（增强前381行）
- 专家来源: 陈次（字节跳动）2024年抖音推荐系统算法
- 状态: 超时，内容已写入，待主编手动确认
```

## 4. 知识工程的评审门控（Quality Gates）

| Gate | 编号 | 检查项 | 通过条件 | 验证方式 |
|------|------|--------|---------|---------|
| G1 规划Gate | G1 | 增强批次是否合理（P0/P1/P2分类） | P0文件≤5个/批 | 主编审查 |
| G2 插入Gate | G2 | 章节是否插入正确位置 | grep确认行号存在 | `rg "章节标题" <file>` |
| G3 规模Gate | G3 | 增强后行数是否≥增强前+50行 | 行数增长 ≥50行 | `wc -l` 对比 |
| G4 来源Gate | G4 | 是否有专家来源标注 | grep "增强来源：" 确认 | `rg "增强来源：" <file>` |
| G5 公式Gate | G5 | 是否含≥1个LaTeX公式或量化表格 | 内容审查 | 主编/Verifier目检 |

### Gate执行顺序

```
G1（规划） → G2（插入） → G3（规模） → G4（来源） → G5（公式）
   ✓           ✓            ✓           ✓           ✓
  通过        通过          通过         通过         通过
```

### 未通过Gate处理

| Gate | 未通过原因 | 处理方式 |
|------|---------|---------|
| G2 | 章节未插入 | 主编手动插入或重新spawn子代理 |
| G3 | 行数增长不足 | 追加内容至达标 |
| G4 | 无来源标注 | 补写"增强来源："段落 |
| G5 | 无公式/表格 | 追加量化公式或数据表格 |

## 6. 主编复盘要点

1. **批次规划先于Spawn**：不要在规划完成前并行spawn，避免依赖混乱
2. **超时不等于失败**：子代理超时后，主会话立即检查文件内容，大多数时候内容已写入成功
3. **行数是规模指标，不是质量指标**：行数达标仅代表规模足够，内容质量由G4/G5 Gate保证
4. **专家来源标注是必选项**：没有来源的知识≈无根之木，G4 Gate必须通过
5. **备选主会话很重要**：每个并行批次在主会话保留备选监控，防止子代理失控
