# Prompt工程专家知识库

> 来源：Prompt Engineering Institute + OpenAI Best Practices · 2026-05-24

---

## 1. 身份定义

### 角色ID
`prompt-engineer`

### 核心职责
设计、测试、迭代Prompt，实现目标任务的稳定高效执行。

### 适用场景
- 需要LLM执行特定任务（分类、提取、生成、推理）
- Prompt输出格式不稳定，需要结构化
- Zero-shot效果差，需要Few-shot引导
- 多轮对话中上下文管理问题
- Chain-of-Thought推理需求

### 不适用场景
- 纯闲聊（无需工程化Prompt）
- 简单问答（直接提问即可）
- 实时性要求极高（Prompt迭代有延迟）
- 涉及隐私/安全的 Prompt 不能外传

## 2. 核心方法论：ACE Framework

### Analyze（分析）
- 明确任务类型：分类/生成/推理/提取/对话
- 识别输出约束：格式/长度/风格/语气
- 列出边界情况（edge cases）
- 识别需要避免的陷阱

### Construct（构建）
- 从简单Prompt开始，逐步增加复杂性
- 使用清晰的结构：指令（Instruction）、上下文（Context）、示例（Example）、输出格式（Output Format）
- 添加必要约束（constraints）
- 考虑版本管理

### Evaluate（评估）
- 多样本测试（≥20个测试用例）
- 覆盖正常场景 + 边界场景
- 测量成功率/一致性
- 记录失败模式

## 3. 关键原则

### 原则1：指令优先
```
✅ 好的：总结以下文本为3个要点，每个要点不超过20字
❌ 差的：请给我总结一下好吗？随便总结一下
```

### 原则2：结构化上下文
```
✅ 好的：
[任务]：判断评论情绪
[评论]：{comment}
[输出格式]：{"sentiment": "positive|neutral|negative", "confidence": 0.0-1.0}

❌ 差的：判断这个评论是正面还是负面
```

### 原则3：Few-shot > Zero-shot（当任务复杂时）
```
❌ Zero-shot（复杂任务容易失败）：
将评论分类：positive/negative/neutral

✅ Few-shot（引导格式）：
评论：这家店服务太好了！→ 分类：positive
评论：还行吧，没什么特别的 → 分类：neutral
评论：等了一个小时，菜还凉了 → 分类：negative
评论：{新评论} → 分类：
```

### 原则4：Chain-of-Thought 仅在需要推理时使用
```
✅ 触发CoT的场景：
- 数学问题
- 逻辑推理
- 多步决策
- 因果分析

❌ 不触发CoT的场景：
- 简单分类（是/否）
- 格式转换
- 信息提取
- 文本生成
```

### 原则5：角色设定有衰减，使用需谨慎
```
✅ 有效用法：
"你是一位有10年经验的资深律师，擅长合同审查"

⚠️ 衰减场景：
- 对话超过10轮，角色设定效果下降
- 与其他约束冲突时，角色设定可能被忽略
- 简单任务不需要角色设定，增加token浪费

❌ 无效用法：
"你是一个超级聪明的AI助手，什么都能做"
```

## 4. 常见错误/陷阱

### 陷阱1：Prompt太长导致质量下降
- LLM对开头和结尾的内容记忆最强
- 中间内容容易被"稀释"
- 解决：压缩Prompt，保留核心指令

### 陷阱2：约束条件互相冲突
```
❌ 冲突示例：
"写一篇专业报告，但要用幽默风格，同时保持严谨学术态度"

→ LLM无法同时满足，造成输出混乱

✅ 解决：明确优先级，或拆分为多个Prompt
```

### 陷阱3：Output Format过于复杂
```
❌ 复杂格式（容易出错）：
{"result": {"labels": [{"name": "标签1", "score": 0.95, "reasons": ["原因1", "原因2"]}]}}

✅ 简单格式（更稳定）：
positive, negative, neutral
```

### 陷阱4：Temperature设置不当
| 场景 | Temperature | 说明 |
|------|-------------|------|
| 分类/提取 | 0.0~0.1 | 确定性输出 |
| 结构化生成 | 0.1~0.3 | 少量变化 |
| 创意写作 | 0.7~0.9 | 随机性高 |
| 代码生成 | 0.0~0.2 | 确定性强 |

### 陷阱5：测试样本偏差
- 用"完美"样本测试 → 实际使用时遇到边缘case失败
- 解决：加入对抗性样本（adversarial examples）

### 陷阱6：未版本管理
- Prompt迭代后无法回滚
- A/B测试无法进行
- 解决：记录Prompt版本与测试结果

## 5. Chain-of-Thought 与 Tree-of-Thought

### Chain-of-Thought（CoT）
```
适用：数学、逻辑、多步推理

格式：
问题：{problem}
解决步骤：
1. [第一步分析]
2. [第二步分析]
3. [第三步分析]
答案：{final_answer}
```

### Tree-of-Thought（ToT）
```
适用：需要探索多个解决方案的场景

格式：
问题：{problem}

分支1（乐观）：[推理链] → 结论A
分支2（中性）：[推理链] → 结论B
分支3（悲观）：[推理链] → 结论C

评估：比较三个分支的可行性，选择最优
```

### 触发条件
| 任务类型 | 建议方法 |
|---------|---------|
| 单步推理/分类 | Zero-shot（直接问） |
| 2-3步推理 | CoT |
| 复杂决策/多方案选择 | ToT |
| 创意发散 | 不使用CoT/ToT |

## 6. 结构化输出（JSON Mode）

### OpenAI JSON Mode
```
使用说明：
- 设置 response_format: {"type": "json_object"}
- Prompt中明确 JSON 结构
- 在系统Prompt中说明"不遵守非JSON内容"
```

### 正则提取（当JSON Mode不可用时）
```
场景：模型输出自由文本，需要提取结构化信息

技巧：
1. 在Prompt中明确输出格式模板
2. 使用正则表达式提取：(\d+)年|(\d+)月
3. 预设解析失败时的默认值
```

### 输出格式设计原则
1. 层级不超过3层
2. 字段名使用英文或中文拼音（避免歧义）
3. 必填字段标记，optional字段说明
4. 避免嵌套数组（容易格式错误）

## 7. 版本管理与迭代策略

### Prompt版本命名
```
v1.0 初始版本
v1.1 修复边界case
v2.0 重大架构调整
```

### 迭代流程
```
测试结果差 → 分析失败模式 → 修改Prompt → 重新测试
     ↑                                      ↓
     ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ←
```

### A/B测试
```
同时运行两个版本：
- Prompt A（控制组）：当前版本
- Prompt B（实验组）：修改版本

至少100个样本，比较：
- 成功率
- 平均响应长度
- Token消耗
```

## 8. 与其他角色的协作

### Handoff Format
```
[当前状态]
- Prompt版本：v1.2
- 测试样本数：50个
- 成功率：82%
- 主要失败场景：{描述}

[已完成]
- 核心指令设计
- Few-shot示例选择
- Output Format定义

[待解决]
- 边界case：包含专业术语的评论（召回率仅60%）
- 需要专家知识：金融行业术语词典

[需要的输入]
- 金融行业术语词典（优先）
- 更多金融评论样本（用于扩充测试集）

[下一步]
- 优化专业术语识别
- 预期提升：召回率 60% → 85%
```

### 前置条件
- 明确的任务目标（非模糊需求）
- 至少10个测试样本（正常场景）
- 明确输出格式要求

### 后置交付物
- Prompt版本文件（v1.x.md）
- 测试结果报告
- 失败case分析
- 建议的后续优化方向
