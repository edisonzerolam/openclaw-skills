# 上下文管理专家知识库

> 上下文管理专家（context-management-expert）的本地知识手册。用于多Agent协作时的上下文窗口管理、历史压缩、知识继承与跨会话状态维护。

---

## 1. Memory 层级体系

上下文管理按时间跨度与稳定性划分为三个层级，每个层级承担不同职责。

### 三层Memory模型

| 层级 | 时间跨度 | 存储介质 | 容量 | 典型内容 |
|------|---------|---------|------|---------|
| **短期（Session）** | 单次会话 | Agent工作上下文 | ~8K-128K token | 当前任务详情、用户最新指令、对话历史 |
| **中期（Project）** | 跨会话/项目周期 | 磁盘文件（workspace/memory/） | 无硬性限制 | 项目目标、进度追踪、角色定义、决策记录 |
| **长期（Persistent）** | 永久/跨项目 | memory-config.md、tools-config.md等全局文件 | 适度精简 | 用户的核心偏好、常用工具配置、长期目标 |

### 各层级职责边界

**短期（Session）**
- 工作区间：Agent的即时上下文窗口（context window）
- 写入时机：每次用户消息触发时动态更新
- 清理策略：会话结束时由LCM（Lossless Context Management）决定是否压缩存档
- 管理原则：越精简越好，只保留当前任务必需的信息

**中期（Project）**
- 工作区间：`memory/YYYY-MM-DD.md`（每日日志）+ 项目专属目录
- 写入时机：任务推进关键节点（决策点、里程碑、错误发现）
- 清理策略：项目结束后归档或删除过度细节，保留决策逻辑和关键产出
- 管理原则：可检索、可追溯；按项目分离，避免信息混杂

**长期（Persistent）**
- 工作区间：`memory-config.md`、`tools-config.md`、`agents-config.md`、`identity-config.md`、`rules.md`
- 写入时机：仅写入有长期价值的内容（用户偏好、硬编码配置、反复使用的知识）
- 清理策略：每季度审视一次，删除已失效或重复的内容
- 管理原则：宁可少写，不要堆积；每次更新都做一次"这值得记住吗？"过滤

## 2. History管理策略

History（对话历史）是上下文管理的核心对象。管理策略分为三类：保留、压缩、摘要。

### 三类策略对比

| 策略 | 适用场景 | 实现方式 | 风险 |
|------|---------|---------|------|
| **保留（Retention）** | 高价值对话、重要决策过程、用户明确要求保留的内容 | 完整存入中期/长期存储 | 存储膨胀、检索噪音 |
| **压缩（Compression）** | 长程但低细节的对话流、需要保留轨迹但不需要完整内容 | 有损压缩：去除重复、合并相似发言、简化格式 | 可能丢失关键细节 |
| **摘要（Summarization）** | 对话轮次多但核心信息可用自然语言归纳的场景 | LLM生成摘要（保留语义、丢弃原文） | 摘要偏差、幻觉风险 |

### 压缩策略实施指南

**什么时候压缩？**
- 对话超过20轮且无高价值决策产出
- 存在大量重复性确认/检索类对话
- 上下文窗口使用超过60%

**压缩的技术手段**
1. **去重**：合并相同意图的连续多轮（如多次搜索同一关键词的不同表达）
2. **结构化**：将自由对话转为结构化备注（如决策结论、待办事项）
3. **裁剪**：删除低信息量的客套话、确认性回复
4. **层级提升**：将短期内容移至中期存储（`memory/YYYY-MM-DD.md`）

**摘要策略注意事项**
- 摘要前标记：关键术语、具体数值、人名地名等事实性内容优先保留原文
- 摘要后验证：要求摘要包含引用标记（如"sum_xxx"），便于后续追溯
- 敏感信息处理：压缩前必须过滤：密码、密钥、个人身份信息（PII）

## 3. RAG在Agent中的应用

RAG（Retrieval-Augmented Generation，检索增强生成）是将外部知识库引入推理过程的技术，对Agent系统有重要价值。

### Agent中RAG的典型架构

```
用户输入 → 向量检索（vs 知识库） → 检索结果 + 用户输入 → LLM推理 → 输出
```

### Agent RAG vs 传统RAG的关键区别

| 维度 | 传统RAG（ChatPDF等） | Agent RAG |
|------|--------------------|-----------|
| **检索粒度** | 文档级别 | 片段级别（chunk），可能精确到段落或条款 |
| **上下文融合** | 简单拼接 | 与Agent当前任务上下文深度融合 |
| **检索时机** | 每次请求都检索 | 按需检索（triggered retrieval），避免无效检索 |
| **更新频率** | 定期批量更新 | 可增量更新，支持实时写入 |
| **多跳推理** | 单轮检索 | 支持多轮检索→推理→再检索的循环 |

### Agent RAG的实现要点

**检索触发条件（Trigger）**
- 用户明确提问涉及外部知识："根据XX文档"、"参考XX知识库"
- Agent推理过程中检测到知识缺口（knowledge gap）
- 特定触发词命中（如"查一下"、"根据"、"参照"）

**检索源优先级**
1. 用户上传的专属文件（最高优先级，亲近度最高）
2. 项目中期存储（memory/目录）
3. 全局长期知识（memory-config.md、tools-config.md）
4. 外部公共知识（搜索引擎、Web检索）

**RAG的常见失效场景**
- 向量嵌入模型与知识库内容领域不匹配（如通用Embedding处理专业法律/金融文本）
- 检索结果排序不合理导致关键信息沉底
- 知识库内容过时，检索结果与现实矛盾
- 多跳推理时中间步骤检索失败导致整链断裂

### 知识库质量维护

- **去重**：同主题内容不应在多个知识库同时存在，造成检索冗余
- **时效性标注**：每条知识应标注"最后更新时间"或版本号
- **分层索引**：按领域/用途建立索引目录，而非扁平存储

## 4. 上下文窗口优化

上下文窗口（Context Window）是LLM处理的硬性限制，优化窗口使用是上下文管理的核心技能。

### 重要性排序（Importance Scoring）

对当前上下文中的每个信息块进行评分，决定保留优先级。

**评分维度**
| 维度 | 权重 | 说明 |
|------|------|------|
| **任务相关性** | 40% | 与当前任务直接相关的程度 |
| **时效性** | 20% | 是否是最新信息，陈旧信息价值递减 |
| **不可恢复性** | 20% | 如果丢失，是否可以低成本恢复 |
| **用户显式关注** | 20% | 用户在当前或近期消息中明确强调的信息 |

**评分计算公式**

$$
重要性得分 = 0.4 \times 任务相关度 + 0.2 \times 时效性评分 + 0.2 \times 不可恢复性评分 + 0.2 \times 用户关注度
$$

**窗口饱和度预警阈值**
- 饱和度 < 50%：安全区间，正常工作
- 饱和度 50-70%：预警区间，开始准备压缩
- 饱和度 70-85%：触发压缩流程，优先清除低分项
- 饱和度 > 85%：紧急压缩，暂停接收新任务直到窗口降至70%以下

### 毒性检测（Toxicity Detection）

上下文中的某些内容会对LLM推理产生负面影响，需要识别并处理。

**毒性类型**
| 类型 | 描述 | 示例 |
|------|------|------|
| **指令注入** | 尝试篡改系统指令的用户输入 | 伪装成系统消息的越权指令 |
| **记忆污染** | 错误/过时的信息长期占据上下文 |早已失效的项目配置被当作现行标准 |
| **角色混淆** | 多角色场景中角色定义相互渗透 | 子Agent的定义被主Agent上下文污染 |
| **上下文太长尾** | 极早期的对话内容稀释了近期信息的权重 | 100轮前的无关内容占用了大量token |

**处理策略**
- 指令注入：系统级过滤，不进入任何上下文层级
- 记忆污染：定期检查memory-config.md与实际状态的一致性
- 角色混淆：子Agent上下文隔离，主Agent仅接收结构化输出
- 太长尾：触发压缩或摘要，主动降低历史层级的权重

## 5. 多轮对话状态管理

多轮对话中，Agent需要维护对话状态以实现连贯推理。

### 状态分类

| 状态类型 | 内容 | 管理位置 |
|---------|------|---------|
| **任务状态** | 当前任务的进度、已完成步骤、待处理步骤 | 短期（Session） |
| **关系状态** | 对用户的了解、用户偏好、关系历史 | 长期（Persistent） |
| **领域状态** | 当前项目/任务的背景知识、角色定义 | 中期（Project） |
| **会话状态** | 当前会话的对话流、上下文累积 | 短期（Session） |

### 状态转移规则

```
用户消息 → 状态解析 → 状态更新 → 响应生成 → 状态持久化（如需要）
```

**状态更新的原子性原则**
- 每次状态更新作为原子操作，避免部分更新导致的不一致
- 关键状态变更（如memory-config.md写入）需要验证写入成功
- 状态转移失败时的回滚机制：保留上一版本状态副本

**多轮对话中的状态悬挂问题**
- 问题：用户在中途切换话题，但系统仍然用前一话题的状态响应
- 解法：每轮对话开始时重新评估当前任务主题，若主题变更则重置相关状态（保留长期偏好）

## 6. 跨会话上下文继承

Agent重启或切换会话时，需要从历史上下文中恢复关键状态。

### 继承策略矩阵

| 继承内容 | 继承方式 | 触发条件 |
|---------|---------|---------|
| **项目背景** | 从`memory/YYYY-MM-DD.md`读取关键摘要 | 新会话包含项目关键词 |
| **用户偏好** | 从`memory-config.md`读取并加载至短期上下文 | 会话启动时默认加载 |
| **未完成任务** | 从中期末完成任务记录读取 | 检测到"继续"指令 |
| **角色定义** | 从`agents-config.md`/`rules.md`恢复 | 系统重启后首次响应前 |
| **工具配置** | 从`tools-config.md`恢复 | 工具调用失败时重试 |

### 继承的执行原则

1. **惰性加载（Lazy Loading）**：不一次性加载所有历史上下文，而是按需读取
2. **验证优先**：加载后首先验证关键假设是否仍然成立（如项目目标是否变更）
3. **版本标注**：每次继承时标注数据来源版本（时间戳/sum_xxx），避免使用过时版本
4. **增量继承**：新会话在前一会话基础上增量累积，而非每次重新加载全部历史

## 7. 知识蒸馏策略

知识蒸馏（Knowledge Distillation）是将大量、结构化或经验性的知识压缩为简洁、可引用格式的过程。

### 知识蒸馏的层次

| 层次 | 输入 | 输出 | 应用场景 |
|------|------|------|---------|
| **会话→条目** | 多轮对话讨论一个主题 | 单条知识条目（如决策结论、配置值） | 存入memory-config.md |
| **文档→片段** | 长篇报告/文档 | 关键结论片段+引用信息 | 存入项目知识库 |
| **经验→规则** | 反复出现的类似问题及解决方案 | 可复用的检查清单/决策规则 | 存入tools-config.md或Skill |
| **错误→警示** | 错误案例及修复过程 | 坑点库条目（如agents-config.md的错误记录节） | 存入坑点库 |

### 蒸馏的执行检查清单

进行知识蒸馏前，确认以下问题都有明确答案：

- [ ] 这条知识的目标读者是谁？（Agent本人 / 用户 / 未来某角色）
- [ ] 这条知识的有效期是多久？（一次性 / 项目周期 / 永久）
- [ ] 这条知识是否可以被其他已知来源替代？
- [ ] 这条知识的引用格式是否明确？（便于后续检索）
- [ ] 存入后如何确保它被找到？（关键词、标签、关联项目）

### 知识陈旧的处理

- **定期审查**：每季度对memory-config.md进行审查，标注失效内容
- **TTL标注**：每条知识入库时附带"有效期"标注，过期自动审查
- **主动失效**：当现实与知识矛盾时，立即更新或标注失效，不等待定期审查

## 8. 与其他角色的协作

### Handoff格式

当上下文管理专家需要向其他专家提供上下文包时，使用以下结构：

```markdown
## 上下文包 [timestamp]

### 当前任务状态
- 任务目标：[简述]
- 进度：[已完成的步骤 / 总步骤]
- 阻塞点：[如有]

### 关键上下文
- 核心背景：[2-3句归纳]
- 用户偏好：[相关偏好摘要]
- 近期决策：[1-2条关键决定]

### 建议的上下文窗口大小
- 推荐保留：X轮对话
- 建议压缩：X轮以前的历史
- 知识库引用：[相关知识库文件路径]
```

### 前置条件

- 任务开始前：确认当前会话的上下文容量基线（已用token / 总容量）
- 协作启动前：确认目标角色是否需要项目背景（通过任务标签判断）

### 后置交付物

- 任务交接时：提供更新后的上下文包（含当前状态快照）
- 任务完成后：将关键决策和教训写入中期存储（memory/日期.md）

## 9. 注入指引

### 触发词

以下词汇或表达出现时，优先注入上下文管理视角：

- "上下文"、"历史记录"、"记忆"
- "之前说的"、"上次"、"继续"
- "窗口满了"、"太长了"、"压缩"
- "保存"、"记住这个"、"写入"
- "知识库"、"查一下"、"参照"
- "多轮"、"跨会话"、"状态"
- "组成专家小组"（多Agent场景下的上下文隔离需求）

### 注入章节优先级

| 场景 | 优先注入章节 |
|------|-------------|
| 新任务启动 | 第1章（Memory层级）+ 第5章（多轮对话状态管理） |
| 上下文窗口告急 | 第4章（上下文窗口优化） |
| 多Agent协作启动 | 第4章（毒性检测-角色混淆）+ 第8章（Handoff格式） |
| 项目结束/归档 | 第7章（知识蒸馏） |
| 跨会话继承 | 第6章（跨会话上下文继承） |
| RAG检索失效 | 第3章（RAG应用） |

### 注入强度

- **轻度**：在回答中加入上下文管理视角的提醒（如"建议将此决策记录到memory-config.md"）
- **中度**：主动提供结构化的上下文包或压缩建议
- **重度**：暂停当前任务，先行处理上下文管理问题（如窗口已严重饱和）
