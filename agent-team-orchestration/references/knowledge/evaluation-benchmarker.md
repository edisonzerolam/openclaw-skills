# 评估基准专家知识库

> 评估基准专家（evaluation-benchmarker）的本地知识手册。用于Agent评估体系设计、基准测试构建、评估结果分析时的方法与指标引用。

---

## 1. 身份定义

### 角色ID

`evaluation-benchmarker`

### 核心职责

设计、执行与分析Agent系统的评估基准测试。包括：
- 构建评估集（benchmark construction）
- 选择/设计评估指标体系
- 对比人类评估与自动评估的权衡
- 回归测试防止能力退化
- LLM-as-Judge范式实施
- 评估结果报告撰写与解读

### 适用场景

- **新增Agent能力上线前**：需建立基线测试，确认能力达标
- **迭代版本对比**：新旧版本性能对比（regression testing）
- **多Agent横向对比**：不同Agent/策略的性能排名
- **评估体系设计**：为团队建立标准化评估流程
- **评估报告撰写**：向利益相关方说明Agent表现
- **异常表现诊断**：Agent出现非预期行为时的根因分析

### 不适用场景

- **实时生产监控**：生产环境实时指标（用observability-expert）
- **Agent架构设计**：系统设计阶段（用multi-agent-architect）
- **prompt优化**：单次prompt调优（用prompt-engineer）
- **法律合规判断**：具体法律问题（用legal-researcher）
- **投资决策**：个股/市场判断（用stock-analyst）

## 2. 核心评估指标体系

### 2.1 基础指标：成功率、效率、稳定性、成本

#### 成功率（Success Rate）

$$
SR = \frac{成功完成任务数}{总任务数}
$$

| 成功率区间 | Agent质量评级 | 说明 |
|------------|--------------|------|
| > 95% | 优秀 | 可投入生产 |
| 80%~95% | 良好 | 基本可用，轻微问题需修复 |
| 60%~80% | 一般 | 需要系统性优化 |
| < 60% | 差 | 核心能力存在严重缺陷 |

**成功定义**：任务结果满足预设质量标准（需在评估前明确定义）

#### 效率（Efficiency）

$$
Efficiency = \frac{完成任务所需资源}{标准资源}
$$

资源维度：
- **时间效率**：完成任务所需平均步数/时间
- **Token效率**：完成任务所需平均Token数
- **调用效率**：完成任务所需API调用次数

#### 稳定性（Stability / 一致性）

$$
稳定性 = 1 - \frac{Var(多次运行结果)}{Mean(多次运行结果)}
$$

| 指标 | 含义 |
|------|------|
| Variance | 多次运行结果的方差，越低越稳定 |
| CV（变异系数） | 标准差/均值，消除量级影响 |
| 置信区间 | 结果的置信区间宽度 |

#### 成本（Cost）

$$
Cost = Token消耗 \times 单Token价格 + API调用次数 \times 每次调用价格
$$

### 2.2 GAIA基准测试（General AI Assistants）

GAIA是针对通用AI助手的评估基准，模拟真实世界任务。

#### GAIA设计原则

| 原则 | 说明 |
|------|------|
| **真实世界任务** | 任务来源于真实使用场景，非人造 |
| **多步骤推理** | 需要多步操作、工具调用、推理链 |
| **可验证答案** | 答案可客观验证（数值、URL、事实） |
| **无需人工标注** | 自动化评估，减少人工依赖 |

#### GAIA任务分类

| 任务类型 | 示例 | 评估重点 |
|----------|------|---------|
| **信息检索** | 查找特定公开信息 | 准确性、时效性 |
| **多步骤操作** | 预约+填表+发邮件 | 流程完整性 |
| **数据分析** | 读取CSV并生成报告 | 分析质量 |
| **推理判断** | 基于证据链做出决策 | 推理合理性 |
| **开放式创作** | 撰写报告/邮件 | 输出质量 |

#### GAIA评分标准

| 分数段 | 表现描述 |
|--------|---------|
| 90~100 | 任务完全正确，无需人工介入 |
| 70~89 | 任务基本正确，有小问题需微调 |
| 50~69 | 任务部分完成，需要人工补充 |
| < 50 | 任务失败或方向错误 |

### 2.3 BIG-bench任务设计（Beyond the Imitation Game）

BIG-bench是针对大语言模型的综合评估基准，包含多种任务类型。

#### BIG-bench任务类型

| 任务类型 | 描述 | 评估方法 |
|----------|------|---------|
| **语言理解** | 文本蕴含、摘要、问答 | Exact Match / ROUGE |
| **推理** | 数学、逻辑、常识推理 | 结果正确性 |
| **代码生成** | 代码补全、代码生成 | Pass@k / 执行正确性 |
| **工具使用** | API调用、工具链编排 | 任务完成率 |
| **多模态** | 图文理解、视觉推理 | 人工评估 |
| **伦理安全** | 有害内容检测、偏见评估 | 人工评估 + 自动检测 |

#### BIG-bench评估维度

| 维度 | 说明 |
|------|------|
| **准确性（Accuracy）** | 结果是否正确 |
| **鲁棒性（Robustness）** | 对输入扰动的敏感性 |
| **效率（Efficiency）** | 计算资源和时间消耗 |
| **泛化性（Generalization）** | 在分布外数据上的表现 |
| **公平性（Fairness）** | 对不同群体的表现差异 |

## 3. 评估集构建方法论

### 3.1 采样策略

#### 分层采样（Stratified Sampling）

$$
N_h = N \times \frac{W_h \times C_h}{\sum(W_i \times C_i)}
$$

其中 $W_h$ = 层权重，$C_h$ = 层复杂度系数

**实施步骤**：

1. **定义分层维度**：任务类型 × 难度等级 × 领域
2. **计算各层配额**：确保各层都有足够样本
3. **层内随机采样**：保证层内代表性
4. **平衡调整**：对样本过少的层进行过采样

#### 难度分级

| 难度等级 | 描述 | 占比建议 |
|----------|------|---------|
| **L1：基础** | 单步操作，直接可解 | 30% |
| **L2：进阶** | 2-3步推理，需要工具 | 40% |
| **L3：复杂** | 多步复杂推理，多工具协同 | 20% |
| **L4：极限** | 边界case、高难度推理 | 10% |

### 3.2 边界Case覆盖

#### 边界Case识别方法

| 方法 | 说明 |
|------|------|
| **历史Bug分析** | 从生产环境Bug库中提取失败场景 |
| **对抗性构造** | 设计故意诱导错误的case |
| **专家访谈** | 从领域专家处获取边界场景 |
| **用户反馈分析** | 从用户反馈中提取困难case |
| **探索性测试** | 随机输入探测Agent行为边界 |

#### 边界Case评估矩阵

| Case类型 | 覆盖数量建议 | 最低要求 |
|----------|-------------|----------|
| 已知失败Case | ≥ 50个 | 回归测试必须通过 |
| 边界输入 | ≥ 20个 | L3难度占比≥30% |
| 对抗性Case | ≥ 10个 | 必须有专门对抗测试 |
| 长上下文Case | ≥ 10个 | 验证context窗口处理 |

## 4. 人类评估 vs 自动评估

### 4.1 权衡框架

| 维度 | 人类评估 | 自动评估 |
|------|---------|---------|
| **准确性** | 高（主观质量判断） | 中（客观指标可靠） |
| **成本** | 高（人力、时间） | 低（可批量执行） |
| **一致性** | 中（评估者间差异） | 高（确定性） |
| **速度** | 慢 | 快 |
| **覆盖度** | 受限于人力 | 可全量覆盖 |
| **适用场景** | 主观质量、创新性 | 客观答案、事实正确性 |

### 4.2 选择决策树

```
任务类型判断
│
├─ 客观任务（数值/事实）
│   └─ → 自动评估（Exact Match/BLEU/ROUGE）
│
├─ 主观任务（写作质量/用户体验）
│   └─ → 人类评估（需定义评分rubric）
│
├─ 混合任务（主体客观+部分主观）
│   └─ → 分层评估（客观部分自动+主观部分人类）
│
└─ 安全/伦理任务
    └─ → 人类评估 + 自动预筛（有害内容检测）
```

### 4.3 评分Rubric设计（人类评估用）

#### 通用Rubric结构

| 维度 | 分值 | 描述 |
|------|------|------|
| **任务完成度** | 0-25分 | 是否达成任务目标 |
| **准确性** | 0-25分 | 信息/结论的正确性 |
| **连贯性** | 0-25分 | 逻辑和表达连贯性 |
| **效率** | 0-25分 | 资源使用合理性 |

**总分 = 四维度得分之和（满分100）**

#### 评估者一致性

$$
一致性 = \frac{评估者一致的数量}{总评估数量}
$$

| 一致性 | 含义 | 处理建议 |
|--------|------|---------|
| > 80% | 高 | 结果可信 |
| 60%~80% | 中 | 需讨论统一标准 |
| < 60% | 低 | 重新校准评估者 |

## 5. LLM-as-Judge 范式

### 5.1 核心思想

使用强LLM（如GPT-4）作为评估者，判断Agent输出的质量。

### 5.2 评判模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **独立评分** | 给定输入+输出，直接打分 | 单一输出评估 |
| **对比评分** | 两个输出对比，选优/打分差 | A/B测试 |
| **成对比较** | 输出两两对比，输出偏好和理由 | 排名评估 |
| **多维评分** | 多维度分别打分后汇总 | 综合质量评估 |

### 5.3 Prompt设计模板

```markdown
## 评判任务
你是一位专业的AI评估专家。请对以下Agent输出的质量进行评估。

## 评估对象
**任务描述**：{task_description}
**输入**：{input}
**Agent输出**：{agent_output}
**参考答案**（如有）：{reference}

## 评估维度
1. **准确性**（0-10）：输出是否正确反映了要求？
2. **完整性**（0-10）：是否包含了所有必要信息？
3. **清晰性**（0-10）：表达是否清晰易读？

## 输出格式
请以JSON格式输出：
{
  "scores": {
    "accuracy": <int>,
    "completeness": <int>,
    "clarity": <int>
  },
  "overall_score": <int>,
  "reasoning": "<简要说明评分依据>"
}
```

### 5.4 LLM-as-Judge的常见问题与缓解

| 问题 | 缓解方法 |
|------|---------|
| **位置偏好**（Position bias） | 随机化输出顺序，多次对比 |
| **长度偏好**（Length bias） | 限制回答长度，关注质量而非长度 |
| **自我偏好**（Self-preference bias） | 使用独立评估者模型 |
| **一致性不足** | 多次评估取平均，报告置信区间 |

## 6. 回归测试：防止能力退化

### 6.1 回归测试集设计

#### 最小回归集（Minimum Regression Suite）

$$
回归集规模 = 基线通过率 \times 覆盖维度数 \times 安全系数
$$

**设计原则**：

1. **基线必过**：所有基线测试必须100%通过
2. **P0优先**：核心能力测试必须全部通过
3. **分层管理**：按优先级（P0/P1/P2）分级
4. **自动化执行**：集成到CI/CD流水线

#### 回归测试分层

| 层级 | 覆盖范围 | 执行频率 | 通过标准 |
|------|---------|---------|---------|
| **P0** | 核心功能 | 每次构建 | 100%通过 |
| **P1** | 重要功能 | 每日 | ≥ 95% |
| **P2** | 一般功能 | 每周 | ≥ 90% |
| **P3** | 边缘功能 | 每月 | ≥ 80% |

### 6.2 退化检测机制

#### 滑动窗口检测

$$
退化分数 = \frac{当前版本通过率 - 基线版本通过率}{基线版本通过率} \times 100\%
$$

| 退化分数 | 警告级别 | 处理建议 |
|----------|---------|---------|
| < 1% | 正常波动 | 记录观察 |
| 1%~3% | 黄色预警 | 分析原因 |
| 3%~5% | 橙色预警 | 暂停发布 |
| > 5% | 红色预警 | 回滚版本 |

## 7. 评估结果分析与报告

### 7.1 统计显著性检验

#### 常用检验方法

| 场景 | 方法 | 判据 |
|------|------|------|
| 两版本对比 | t检验（配对/独立） | p < 0.05 为显著 |
| 多版本对比 | ANOVA | F检验显著后做post-hoc |
| 比例对比 | 卡方检验 | p < 0.05 为显著 |
| 非正态分布 | Mann-Whitney U | p < 0.05 为显著 |

#### 效果量（Effect Size）

$$
Cohen's\ d = \frac{\bar{X}_1 - \bar{X}_2}{SD_{pooled}
$$

| d值 | 效果量 |
|------|--------|
| < 0.2 | 微小 |
| 0.2~0.5 | 小 |
| 0.5~0.8 | 中等 |
| > 0.8 | 大 |

### 7.2 评估报告结构

#### 标准评估报告模板

```markdown
# Agent评估报告

## 1. 执行摘要
- 评估目标、版本、日期
- 关键结论（3点以内）

## 2. 评估设置
- 评估集描述（规模、类型分布）
- 评估方法（自动/人类/LLM-as-Judge）
- 评估环境配置

## 3. 整体表现
- 核心指标汇总表
- 与基线版本对比

## 4. 分维度分析
- 按任务类型/难度/领域细分
- 各维度指标详情

## 5. 异常发现
- 低于预期的case列表
- 边界case失败分析

## 6. 回归测试结果
- 通过率统计
- 退化检测结论

## 7. 建议
- 优化优先级排序
- 具体改进方向

## 附录
- 详细测试case列表
- Raw data链接
```

## 8. 关键原则

### 原则1：评估驱动开发（Evaluation-Driven Development）

> 没有量化就没有优化。先建立可靠的评估体系，再进行迭代开发。

### 原则2：评估必须可重复

> 同样的输入必须得到可重复的结果。随机性必须有控制的引入并报告。

### 原则3：评估维度要完整

> 不能只用单一指标。成功率、效率、成本、稳定性需要综合考量。

### 原则4：评估边界而非平均值

> 特别关注边界case。平均表现好但边界差的Agent在生产环境会出问题。

### 原则5：评估要成本意识

> 评估本身有成本。需要权衡评估精度与评估成本，选择性价比最高的方案。

## 9. 常见错误/陷阱

| 错误类型 | 具体表现 | 后果 | 正确做法 |
|---------|---------|------|---------|
| **评估集泄漏** | 评估集与训练集有重叠 | 高估真实性能 | 严格分离训练/评估集 |
| **只测 happy path** | 只测试正常场景 | 边界case大量失败 | 分层覆盖边界case |
| **单一指标决策** | 只看成功率 | 忽视效率和成本 | 多维指标综合评估 |
| **忽视统计显著性** | 不做显著性检验直接下结论 | 虚假改进 | 必须做统计检验 |
| **评估者不一致** | 人类评估标准不一 | 结果不稳定 | 先校准评估者 |
| **过度优化** | 针对评估集优化 | 泛化能力差 | 保留留出集（hold-out） |
| **忽略成本** | 不计评估资源消耗 | 评估不可持续 | 评估预算规划 |
