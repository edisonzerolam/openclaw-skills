# 工作流编排专家知识库

> 工作流编排专家（workflow-orchestrator）的本地知识手册。用于多智能体系统的任务编排、DAG 设计、状态机模式应用与长时间运行任务的分段策略设计。

---

## 1. 身份定义

| 属性 | 内容 |
|------|------|
| **角色ID** | `workflow-orchestrator` |
| **核心职责** | 设计、评估、改进多智能体系统的任务编排策略，包括 DAG 结构设计、并行/串行执行决策、条件分支路由、回滚重试机制与工作流版本控制 |
| **适用场景** | 复杂多阶段任务拆解、专家小组协作流程编排、长时间运行任务的阶段划分、Crew/Manager-Crew 模式的任务路由设计、工作流框架选型（Temporal/Airflow/Prefect/LangGraph） |
| **不适用场景** | 单 Agent 内部 Prompt 优化、工具 API 调用细节、具体业务指标的计算实现 |
| **输出形式** | DAG 可视化设计、状态机转换图、任务分解决策表、编排框架对比报告 |

## 2. 核心方法论

### 2.1 DAG（有向无环图）设计原则

DAG 是工作流编排的基础结构。所有任务节点按依赖关系组织，必须保证无环。

**核心原则**：

1. **单入口、单出口**：每个工作流有明确的起点和终点，便于推理执行路径
2. **节点粒度适中**：单个节点完成一个语义完整的子任务，避免将原子操作拆成独立节点
3. **依赖关系显式化**：节点间依赖通过边明确定义，不用隐式状态传递
4. **并行分支独立**：无依赖的节点应并行执行，不应人为串行化
5. **可插拔节点**：节点实现与编排逻辑解耦，同一节点可被不同工作流复用

**DAG 验证 checklist**：

```
□ 是否存在环（循环依赖）？
□ 入口节点是否无父节点？
□ 出口节点是否无子节点？
□ 每个节点是否至少有 1 个父节点（入口除外）？
□ 并行分支是否真的无依赖？
□ 条件分支是否有兜底路径（else/none）？
```

**DAG 表示方式（YAML 示例）**：

```yaml
workflow:
  name: multi-expert-review
  entry: intake
  exit: report_compile

nodes:
  intake:
    role: intake-specialist
    next: [data-gather, search]
  data-gather:
    role: data-specialist
    requires: [intake]
    next: [analysis]
  search:
    role: researcher
    requires: [intake]
    next: [analysis]
  analysis:
    role: analyst
    requires: [data-gather, search]  # 汇聚节点
    next: [review, synthesis]
  review:
    role: auditor
    requires: [analysis]
    next: [synthesis]
  synthesis:
    role: synthesis-writer
    requires: [analysis, review]
    next: [report_compile]
  report_compile:
    role: report-compiler
    requires: [synthesis]
    next: []
```

### 2.2 状态机模式在 Agent 工作流中的应用

单个 Agent 内部可建模为状态机，Agent 与 Agent 间的协作也可整体建模为状态机。

**Agent 内部状态机典型状态**：

| 状态 | 说明 | 可转向前置状态 |
|------|------|--------------|
| `idle` | 等待任务分配 | - |
| `running` | 执行中 | `idle` |
| `waiting_input` | 阻塞等待外部输入/工具结果 | `running` |
| `paused` | 暂停（人力审核中） | `running` |
| `completed` | 正常完成 | `running` |
| `failed` | 执行失败 | `running` |
| `cancelled` | 被取消 | `idle`, `running` |

**状态转换规则**：

```
idle → running：任务分配触发
running → waiting_input：遇到工具调用
waiting_input → running：工具结果返回
running → completed：正常结束
running → failed：超出重试上限或不可恢复错误
running → paused：需要人工审批
paused → running：审批通过
任何非 idle 状态 → cancelled：主动取消
```

**多 Agent 协作状态机示例（简化）**：

```
workflow-state: RUNNING
  ├─ node: intake        → COMPLETED
  ├─ node: data-gather  → COMPLETED
  ├─ node: search       → RUNNING  ← 当前执行节点
  ├─ node: analysis     → PENDING
  ├─ node: synthesis    → PENDING
  └─ node: report       → PENDING
```

### 2.3 并行 vs 串行任务执行的选择

**并行执行的条件**（满足任一即可）：

- 两个节点**互不依赖**：A 节点的输出不影响 B 节点执行，B 节点也不依赖 A 的输出
- 两个节点**访问不同资源**：如同时调用不同 API、读取不同数据源，无竞争
- 需要**结果对比/投票**：如多个分析师独立评估同一标的
- **时间敏感任务**：主流程等待最慢节点，但并行执行可降低总延迟

**必须串行执行的情况**：

- B 节点**依赖 A 节点的输出**
- A 节点的执行结果用于**决定 B 节点是否执行**（条件分支）
- 两个节点访问**同一有限资源**（如数据库写锁、外部 API 限流）
- 需要**人工审核**后继续执行

**决策矩阵**：

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| 多个专家独立分析同一问题 | 并行 | 避免串行等待，加速执行 |
| 数据抓取 → 数据清洗 → 分析 | 串行 | 严格依赖链 |
| 多个数据源并发获取 | 并行 | I/O 密集，无竞争 |
| 分析 → 人工审核 → 报告生成 | 串行 | 人审是不可省略步骤 |
| 同一标的的多个因子并行计算 | 并行 | 计算密集，独立无依赖 |
| 主流程旁路的信息收集 | 并行 | 不阻塞主流程 |

### 2.4 条件分支与动态路由

**条件分支设计原则**：

1. **每个条件分支必须有兜底路径**：不允许任务到达一个无任何后继的节点后悬停
2. **条件表达式需可求值**：条件判断所需的数据在分支决策点必须已就绪
3. **分支数量控制**：建议不超过 5 个分支，否则难以维护

**动态路由常见模式**：

```python
def route_after_analysis(context):
    """分析完成后的路由决策"""
    confidence = context.analysis_result.confidence

    if confidence >= 0.9:
        return "direct_report"          # 高置信度 → 直接生成报告
    elif confidence >= 0.6:
        return "review_then_report"     # 中等置信度 → 追加审核节点
    elif confidence >= 0.3:
        return "re_analyze"             # 低置信度 → 重新分析
    else:
        return "human_escalation"       # 极低置信度 → 升级人工处理
```

**路由决策表**：

| 条件 | 路由目标 | 说明 |
|------|---------|------|
| `confidence >= 0.9` | `direct_report` | 跳过审核，加速流程 |
| `0.6 <= confidence < 0.9` | `review_then_report` | 标准流程 |
| `0.3 <= confidence < 0.6` | `re_analyze` | 重新分析（最多 N 次） |
| `confidence < 0.3` | `human_escalation` | 人工介入 |
| `analysis_error != null` | `error_handling` | 异常处理分支 |

### 2.5 回滚与重试机制

**重试策略设计**：

| 重试策略 | 适用场景 | 不适用场景 |
|---------|---------|-----------|
| **立即重试（指数退避）** | 瞬时网络抖动、短暂服务不可用 | 长期资源枯竭错误 |
| **固定间隔重试** | 外部 API 限流（429）有固定恢复时间 | 高频不确定错误 |
| **手动重试（人工介入）** | 需要外部修复的错误（如凭据失效） | 自动化流程中 |
| **放弃并标记失败** | 不可恢复错误（如数据格式损坏） | 任何需要保留结果的场景 |

**指数退避实现示例**：

```python
def retry_with_backoff(fn, max_attempts=3, base_delay=1.0, max_delay=30.0):
    for attempt in range(max_attempts):
        try:
            return fn()
        except RetryableError as e:
            if attempt == max_attempts - 1:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            time.sleep(delay)
```

**回滚机制设计**（Saga 模式）：

```
Task_A → Task_B → Task_C → Task_D
          ↓         ↓         ↓
       ROLLBACK_D  ROLLBACK_C  -
```

- 每个可回滚步骤注册补偿操作（compensate）
- 任意步骤失败时，按逆序执行已成功步骤的补偿操作
- 补偿操作必须是幂等的

**Agent 工作流重试级别**：

| 级别 | 范围 | 示例 |
|------|------|------|
| 工具调用级 | 单次 API 调用 | HTTP GET 超时 → 重试 3 次 |
| Agent 执行级 | 单个 Agent 单次运行 | 分析 Agent 超时 → 重新调度 |
| 节点级 | 整个节点重新执行 | 数据抓取节点失败 → 重跑整个节点 |
| 工作流级 | 整个工作流重启 | 多节点失败 → 从 checkpoint 重启 |

### 2.6 工作流版本控制

**版本控制核心原则**：

1. **不可变版本**：每个已发布版本的工作流定义不可修改
2. **语义版本号**：`major.minor.patch`（重大重构/功能变更/缺陷修复）
3. **版本亲和**：运行时实例绑定特定版本，不随版本升级自动迁移
4. **版本元数据**：每个版本记录创建时间、创建人、变更说明

**版本迁移策略**：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **直接切换** | 新版本直接替代旧版本，所有新任务使用新版本 | 实验性工作流、内部工具 |
| **蓝绿部署** | 两版本并行，通过路由比例切换 | 生产级工作流，需灰度验证 |
| **金丝雀发布** | 先将小比例任务路由到新版本，稳定后逐步放大 | 高风险变更 |
| **特性开关** | 代码层面通过开关控制版本逻辑 | 需要快速回滚的场景 |

**版本兼容性处理**：

```python
# 版本兼容性检测
def check_version_compatibility(wf_version, node_versions):
    issues = []
    for node, v in node_versions.items():
        required = wf_version.supported_node_versions.get(node)
        if required and not semver_compatible(v, required):
            issues.append(f"Node '{node}' version {v} not compatible with workflow {wf_version}")
    return issues
```

### 2.7 常见编排框架对比

| 框架 | 核心模型 | 优势 | 劣势 | 适用规模 |
|------|---------|------|------|---------|
| **Temporal** | 持久化状态机 + 事件溯源 | 极强的容错能力，支持长时间运行任务，内置重试/回滚 | 运维复杂度高，学习曲线陡 | 中大型生产系统 |
| **Airflow** | DAG 调度 | 生态成熟，监控完善，适合批处理 | 不适合交互式/事件驱动任务 | 数据管道，批处理 |
| **Prefect** | DAG 调度（Python 优先） | Pythonic，学习成本低，云原生 | 相比 Temporal 容错能力弱 | 数据工程，轻量级工作流 |
| **LangGraph** | 状态图 + 条件路由 | 与 LLM 原生集成，支持图结构动态修改，多智能体状态共享 | 成熟度较前两者低 | LLM 应用，多智能体编排 |
| **CrewAI / AutoGen** | Agent 角色 + 任务分配 | 多 Agent 协作抽象简单 | 任务编排灵活性受限 | 快速原型，多 Agent 原型 |

**框架选型决策树**：

```
是否需要长时间运行（>30min）且容错要求高？
├─ 是 → Temporal
└─ 否 → 是否主要是 LLM/多 Agent 应用？
    ├─ 是 → LangGraph / CrewAI
    └─ 否 → 是否需要定期批处理？
        ├─ 是 → Airflow / Prefect
        └─ 否 → 是否需要快速原型？
            ├─ 是 → CrewAI
            └─ 否 → 自定义状态机实现
```

### 2.8 长时间运行任务的分段策略

**分段的本质**：将一个长时间运行的任务拆解为多个可独立执行、可检查点保存、可重试的阶段。

**分段策略原则**：

1. **以 I/O 边界为分界点**：网络请求、文件读写等 I/O 操作后设置检查点
2. **以决策点为分界点**：需要基于中间结果做路由决策的位置天然适合分段
3. **避免在计算中途分段**：部分计算结果无法恢复
4. **每段时长建议**：30 秒 ~ 30 分钟，过短增加协调开销，过长降低容错粒度

**分段执行模型**：

```
Phase_1 (input) ──→ Checkpoint_1 ──→ Phase_2 (process) ──→ Checkpoint_2 ──→ Phase_3 (output)
     ↓                  ↓                  ↓                   ↓                  ↓
  持久化状态        持久化状态         持久化状态          持久化状态         持久化状态
```

**检查点（Checkpoint）设计**：

```python
@dataclass
class WorkflowCheckpoint:
    workflow_id: str
    version: str
    current_phase: str
    completed_phases: List[str]
    phase_outputs: Dict[str, Any]  # 已完成阶段的输出，供后续阶段使用
    context: Dict[str, Any]         # 全局上下文
    timestamp: datetime
```

**超时与续跑**：

| 场景 | 处理方式 |
|------|---------|
| 单阶段超时 | 重试该阶段（有限次），超限则标记阶段失败 |
| 工作流整体超时 | 保留最后一个 checkpoint，触发续跑或人工介入 |
| Agent 无响应（hang） | 每个 Agent 设置心跳超时，超时则强制终止并重启 |
| 进程崩溃 | 重启后从最后一个 checkpoint 恢复，无需从头开始 |

**长时间任务监控指标**：

- `progress_rate`：已完成阶段数 / 总阶段数
- `estimated_time_remaining`：基于历史数据估算剩余时间
- `failure_rate_per_phase`：各阶段失败率，用于识别瓶颈
- `avg_phase_duration`：各阶段平均执行时长，用于调整超时配置

## 3. 关键原则（5 条）

**原则 1：DAG 设计先行，代码实现在后**
在设计工作流时，先用 YAML/JSON/图形明确 DAG 结构，再实现节点逻辑。禁止"边写边加节点"的迭代式 DAG 构建。

**原则 2：状态可查询，异常可追溯**
每个工作流实例在任何时刻的状态（运行中/暂停/完成/失败）应可查询；失败时能追溯到具体节点和错误原因。

**原则 3：并行是默认，串行是特例**
无依赖的任务默认并行执行。只有当存在明确的数据依赖或资源竞争时才串行化。

**原则 4：回滚设计要与正向流程同步设计**
补偿操作（回滚逻辑）必须与正向操作在同一个 PR 中实现，不允许先上线正向流程再补回滚。

**原则 5：工作流版本即契约，变更需评审**
工作流定义文件的变更应经过评审，版本号升级需有变更说明，禁止修改已执行的实例对应的版本定义。

## 4. 常见错误与陷阱

### 4.1 DAG 设计阶段

| 陷阱 | 描述 | 后果 | 规避方法 |
|------|------|------|---------|
| **隐藏依赖** | 两个节点看似独立，实际存在隐式数据依赖（如共享全局状态） | 并行执行时结果不确定 | 强制节点间通过显式输入输出传递数据 |
| **长依赖链** | 10+ 节点的串行链，单节点失败导致整链重启 | 容错能力极差 | 长链中插入检查点，每 3-5 节点设置一个同步点 |
| **星形陷阱** | 所有节点汇聚到单一节点，造成该节点负载过高 | 性能瓶颈，整体延迟高 | 分析节点的扇入扇出比，瓶颈节点并行化或拆分 |
| **缺乏兜底分支** | 条件分支没有 else 分支，条件不满足时任务悬停 | 工作流死锁 | 每个分支决策必须覆盖所有可能情况（含 null/unknown） |

### 4.2 状态机实现阶段

| 陷阱 | 描述 | 后果 | 规避方法 |
|------|------|------|---------|
| **状态爆炸** | 引入过多细粒度状态，状态转换规则难以维护 | 状态机不可维护 | 合并相似状态，保持状态数 ≤ 8 |
| **死状态** | 某些状态无任何可转出的路径 | 任务无法继续，卡死 | 定期审查状态转换图的连通性 |
| **幽灵状态** | 代码中存在但状态图中未记录的状态 | 逻辑与图不一致 | 状态图与代码一一对应，代码审查时同步检查状态图 |
| **条件竞争** | 多个actor可同时触发同一状态转换 | 状态不可预测 | 使用互斥锁或单actor控制状态转换 |

### 4.3 执行阶段

| 陷阱 | 描述 | 后果 | 规避方法 |
|------|------|------|---------|
| **节点内重试覆盖关键错误** | 将不可恢复的错误（如格式损坏）当作可重试错误处理 | 无限重试循环 | 区分 RetryableError 和 FatalError，前者重试，后者直接失败 |
| **无检查点的长任务** | 长时间运行任务崩溃后必须从头重跑 | 资源浪费，延迟高 | 任何超过 5 分钟的任务必须设置检查点 |
| **幽灵节点** | 工作流定义中包含但从未被调用的节点 | 维护混乱 | CI 检查工作流 DAG 覆盖度，孤立节点报警 |
| **版本漂移** | 工作流定义被修改但未升级版本号 | 问题追溯困难 | 版本号自动化管理，修改触发强制版本升级校验 |

## 5. 与其他角色的协作

### 5.1 Handoff Format（交接格式）

```markdown
## 工作流交接报告

**工作流名称**：[name]
**版本**：[version]
**当前阶段**：[current_phase]
**执行状态**：[running|paused|completed|failed]

### 已完成节点
- [node_name]: [status] — [简短输出描述]

### 当前节点状态
- **节点名**：[node_name]
- **状态**：[idle|running|waiting|completed|failed]
- **已运行时间**：[duration]
- **输出预览**：[关键字段预览]

### 上下文传递
**传递给下一节点的数据**：
- `[key]`: [value] — [说明]

### 异常记录（如有）
- **[error_node]**: [error_type] — [简短描述]
- 已执行补偿操作：[compensation_action]

### 下一步建议
- [next_action_1]
- [next_action_2]
```

### 5.2 前置条件

接收工作流编排任务前，需确认以下信息已就绪：

- [ ] 工作流的**输入定义**（哪些参数是必需的）
- [ ] 各节点的**实现是否可用**（或明确的占位符）
- [ ] **节点间的依赖约束**（哪些节点必须先于其他节点完成）
- [ ] **异常处理要求**（哪些节点允许重试，哪些必须直接失败）
- [ ] **超时与监控需求**（每个节点的超时配置）

## 6. 注入指引

### 6.1 触发词

当任务请求中出现以下词汇或意图时，注入本角色：

| 触发词 | 匹配场景 |
|--------|---------|
| **编排**、**工作流**、**流程设计**、**DAG** | 明确要求设计或评审工作流结构 |
| **并行**、**串行**、**依赖**、**任务分配** | 涉及任务执行顺序的决策 |
| **状态机**、**状态转换** | 需要建模 Agent 或工作流状态 |
| **超时**、**重试**、**回滚**、**容错** | 涉及任务可靠性设计 |
| **Temporal**、**Airflow**、**Prefect**、**LangGraph** | 框架选型讨论 |
| **长时间任务**、**分段**、**检查点** | 需要设计任务分段策略 |
| **版本**、**升级**、**迁移** | 涉及工作流版本管理 |

### 6.2 注入章节优先级

| 优先级 | 注入章节 | 场景 |
|--------|---------|------|
| **P0** | §1 身份定义 + §2.1 DAG 设计原则 | 任何工作流设计任务 |
| **P1** | §2.3 并行/串行决策 + §2.4 条件路由 | 涉及任务分配或分支决策 |
| **P2** | §2.5 回滚重试 + §4 常见陷阱 | 容错或可靠性设计讨论 |

- 工作流编排是**元级别设计**，不深入具体业务逻辑
- 涉及多角色协作时，先注入 `multi-agent-architect` 的角色知识，再从其框架中细化编排策略
- 遇到状态机设计瓶颈时，可引入 `error-recovery-specialist` 协助异常处理设计
