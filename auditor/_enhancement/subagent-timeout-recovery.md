# 增强层B：子代理超时恢复模式

> 本增强层基于 ClawTeam 知识库增强实战中 3组子代理全部超时（超时率100%）的经验沉淀，用于指导 auditor 在子代理超时场景下的标准化恢复流程和预防策略。

---

## B1. 超时模式识别

### B1.1 超时场景分类

子代理超时按任务类型分为以下两类，识别逻辑不同：

| 超时类型 | 任务特征 | 超时原因 | 恢复难度 |
|---------|---------|---------|---------|
| 知识增强类 | 内容写入，复杂度与内容长度正相关 | 写入内容量大（50~80行/项） | 中等（写入成功率高） |
| 调研类 | 信息收集+写入，输出不确定性高 | 信息源不稳定/格式转换复杂 | 较高（输出不完整） |

### B1.2 本次实战超时案例

```
本次实战超时记录：

第1组子代理（超时）：
- 任务：stock-analyst.md + macro-forecasting.md 双文件增强
- 超时时间：5分钟（300秒）
- 执行状态：子代理执行超时，但文件写入可能已完成
- 恢复方式：主会话备选直接输出

第2组子代理（超时）：
- 任务：hk-stock.md + alternative-data.md 双文件增强
- 超时时间：5分钟（300秒）
- 执行状态：子代理执行超时
- 恢复方式：主会话备选直接输出

第3组子代理（超时）：
- 任务：valuation-expert.md 单文件多增强项
- 超时时间：5分钟（300秒）
- 执行状态：子代理执行超时
- 恢复方式：主会话备选直接输出

超时率统计：
- 子代理执行：3组
- 超时：3组
- 超时率：100%（远超60%阈值）
- 结论：批次划分颗粒度问题（单批次文件数过多）
```

### B1.3 超时风险识别规则

```
高超时风险任务识别（auditor 行动项）：

风险识别关键词检测（任务描述中包含以下词→高风险）：
[ ] "调研" → 调研类任务，超时风险⭐⭐⭐⭐
[ ] "增强" → 知识增强类任务，超时风险⭐⭐⭐
[ ] "调研+写入" → 复合任务，超时风险⭐⭐⭐⭐⭐
[ ] "多文件" → 文件数>3，超时风险⭐⭐⭐⭐
[ ] "专家理论" → 需要信息收集，超时风险⭐⭐⭐⭐

风险等级：
- 高风险（⭐⭐⭐⭐⭐）：调研+多文件 → 必须拆分≤2文件/批次
- 中风险（⭐⭐⭐⭐）：调研单文件/多文件增强 → 拆分≤3文件/批次
- 低风险（⭐⭐⭐）：单文件增强 → 可接受4文件/批次

本次实战案例：
- "stock-analyst.md + macro-forecasting.md + hk-stock.md + alternative-data.md 4文件并行增强"
  → 关键词：增强、多文件
  → 风险等级：⭐⭐⭐⭐（中风险）
  → 应拆分≤3文件/批次，但实际用了4文件 → 超时
```

## B2. 超时预防策略（E5备份）

### B2.1 任务分解规则

```
E5备份策略：任务分解是超时预防的核心

分解规则：
- 超过3个文件的任务 → 必须拆分为≤2文件/子代理
- 单文件多增强项（>3项）→ 拆分为单增强项子任务
- 复合任务（调研+写入）→ 拆分为独立调研+独立写入

分解决策树：
```
开始 → 文件数>3？ → 是 → 拆分≤2文件/批次
     ↓ 否
     文件数>1？ → 是 → 单批次≤3文件
     ↓ 否
     单文件增强项>3？ → 是 → 拆分增强项
     ↓ 否
     可接受4文件/批次
```

本次实战错误复盘：
- 任务：4文件并行增强（stock-analyst + macro-forecasting + hk-stock + alternative-data）
- 错误：未拆分，单批次4文件
- 结果：超时率100%
- 正确做法：拆分为2组（每组≤2文件）

正确批次划分示例（修正后）：
```
P0批次（修正）：
  子代理A1：stock-analyst.md（单文件，3项增强）
  子代理A2：macro-forecasting.md（单文件，2项增强）
  子代理B1：hk-stock.md + alternative-data.md（2文件，2项增强）
```
```

### B2.2 超时阈值配置

```
超时阈值配置：根据任务类型差异化设置

标准阈值（不区分任务类型）：
- 5分钟（300秒）← 默认配置，不适合知识增强类任务

调整后阈值：
- 知识增强类 → 3分钟（180秒）
  原因：内容写入时间可预估，50~80行约需2~3分钟
- 调研类 → 2分钟（120秒）
  原因：信息收集不确定性高，超时后主会话备选更高效
- 复合任务（调研+写入）→ 2分钟 + 主会话备选
  原因：超时即失败，不值得等待

阈值配置示例（agent-team spawn 命令）：
```bash
agent-team spawn subprocess --profile openclaw-wsl \
  --timeout 180 \  # 知识增强类3分钟
  --task '...' \
  --fallback '主会话备选方案'

agent-team spawn subprocess --profile openclaw-wsl \
  --timeout 120 \  # 调研类2分钟
  --task '...' \
  --fallback '主会话备选方案'
```
```

### B2.3 主会话备选方案配置

```
主会话备选方案：每个并行批次必须配置

备选方案设计原则：
1. 备选内容应完整覆盖子代理任务目标
2. 备选内容应可直接执行（主会话接管）
3. 备选触发条件：子代理超时或执行失败

备选方案模板：
```
子代理任务：{任务描述}
子代理超时方案：
  - 触发条件：子代理执行时间超过{阈值}秒
  - 主会话操作：直接执行以下任务
  - 备选执行内容：
    1. 读取目标文件
    2. 识别增强插入位置
    3. 写入增强内容
    4. 验证增强结果
```

本次实战备选方案配置示例：
```
子代理任务：stock-analyst.md + macro-forecasting.md 双文件增强
子代理超时方案：
  - 触发条件：子代理执行时间超过180秒
  - 主会话操作：主会话直接接管执行
  - 备选执行内容：
    1. 读取 stock-analyst.md
    2. 注入护城河分析、估值倍数、宏观配置 三项增强
    3. 读取 macro-forecasting.md
    4. 注入美林时钟、宏观因子 二项增强
    5. grep验证行号
```

备选方案触发后：
- 记录到审计日志（超时次数+恢复方式）
- 评估是否需要调整批次大小
- 更新memory-config.md超时统计
```

## B3. 超时后恢复流程

### B3.1 恢复步骤总览

```
超时后恢复流程（6步法）：

步骤1：检查子代理输出状态
步骤2：检查目标文件是否已写入（部分成功检测）
步骤3：grep目标章节标题（执行后超时但写入成功情况）
步骤4：评估超时原因（任务颗粒度 vs 模型问题）
步骤5：重新spawn单个文件任务（避免连锁超时）
步骤6：验证新任务完成
```

### B3.2 详细恢复步骤

```
步骤1：检查子代理输出状态
操作：查看子代理的最后输出日志
预期：返回"超时"或"执行中"状态
命令示例：
```bash
agent-team status --agent <agent-id>
# 或查看 exec 历史
```

步骤2：检查目标文件是否已写入
操作：检查文件修改时间+内容完整性
预期：文件修改时间在超时时间之后
命令示例：
```bash
# 检查文件修改时间
Get-Item "references/knowledge/a-share-analysis/stock-analyst.md" | Select-Object LastWriteTime

# 检查文件内容（行数）
wc -l references/knowledge/a-share-analysis/stock-analyst.md
```

步骤3：grep目标章节标题
说明：子代理可能在超时前已完成写入，仅回复超时
命令示例：
```bash
grep -n "## 3.2 护城河分析" references/knowledge/a-share-analysis/stock-analyst.md
# 结果：123行  ## 3.2 护城河分析
# 说明：写入成功，仅超时未返回
# 处理：标记为"超时但成功"，无需重做
```

步骤4：评估超时原因
判断标准：
- 超时+文件已写入 → 任务颗粒度问题（批次太大）
- 超时+文件未写入 → 模型问题/任务复杂度问题
- 部分写入 → 任务复杂度问题

步骤5：重新spawn单个文件任务
说明：避免连锁超时，单文件任务更可控
命令示例：
```bash
agent-team spawn subprocess --profile openclaw-wsl \
  --agent-name stock-enhancer-single \
  --task '为 stock-analyst.md 注入护城河分析增强（单文件）' \
  --timeout 180
```

步骤6：验证新任务完成
验证项：
[ ] grep章节标题存在
[ ] wc行数增长符合预期
[ ] grep来源标注存在
```

### B3.3 恢复流程决策树

```
超时恢复决策树：

子代理超时？
  ↓ 是
检查文件是否已写入？
  ↓ 已写入
标记"超时但成功"
记录到审计日志
无需重做
  ↓ 未写入
grep章节标题存在？
  ↓ 存在
标记"超时但成功（章节存在）"
记录到审计日志
无需重做
  ↓ 不存在
评估超时原因
  ↓
原因=批次太大？
  ↓ 是 → 拆分≤2文件/批次
重新spawn单个文件任务
  ↓ 否
原因=模型问题？
  ↓ 是 → 调整模型/增加超时阈值
重新spawn任务
  ↓ 否
原因=任务复杂度？
  ↓ 是 → 拆分增强项
重新spawn任务
```

## B4. 超时率监控（S5.7增强层追踪）

### B4.1 超时率监控指标

```
S5.7增强层追踪：超时率监控指标定义

核心指标：
1. 单批次超时率 = 超时子代理数 / 批次子代理总数
2. 累计超时任务数 = 审计周期内超时任务总数
3. 超时恢复成功率 = 超时后成功恢复数 / 超时总数

阈值定义：
- 单批次超时率 > 60% → 触发「任务复杂度重新评估」
- 累计超时任务数 > 5 → 触发「超时根因分析」
- 超时恢复成功率 < 80% → 触发「恢复流程优化」

本次实战监控数据：
- P0批次：3个子代理，超时3个，超时率100% → 触发重新评估
- P1批次：调整后，6个子代理，超时2个，超时率33% → 通过
- P2批次：调整后，4个子代理，超时0个，超时率0% → 通过
- 累计超时任务数：5个 → 触发根因分析
- 超时恢复成功率：100%（主会话备选全部成功）→ 良好
```

### B4.2 超时根因分析方法

```
超时根因分析（累计超时>5时触发）：

根因分析维度：
1. 任务颗粒度问题（最常见）
   - 单批次文件数过多
   - 单文件增强项过多
   - 解决：拆分任务

2. 模型问题（较少见）
   - 模型响应速度慢
   - 解决：调整模型或增加超时阈值

3. 任务复杂度问题
   - 任务描述模糊导致模型反复确认
   - 解决：优化任务描述

根因分析步骤：
步骤1：收集超时任务列表（时间戳+任务描述+文件数+增强项数）
步骤2：分类超时原因（颗粒度/模型/复杂度）
步骤3：计算各原因占比
步骤4：输出根因报告+改进建议

本次实战根因分析结果：
```
超时根因分析报告：
- 分析周期：2026-05-17 P0批次执行
- 超时任务：3个
- 根因分布：
  1. 任务颗粒度问题：3个（100%）
  2. 模型问题：0个（0%）
  3. 任务复杂度问题：0个（0%）
- 根因结论：单批次4文件导致超时（应≤3文件）
- 改进建议：下次批次划分≤3文件/批次
```
```

## B5. 超时恢复模板（auditor 快速使用）

### B5.1 超时检查脚本

```powershell
# 超时检查脚本（Windows PowerShell）
# 用法：保存为 check-timeout.ps1，执行 .\check-timeout.ps1

param(
    [string]$FilePath,
    [string]$SectionTitle,
    [int]$TimeoutSeconds = 300
)

Write-Host "=== 超时检查脚本 ===" -ForegroundColor Cyan
Write-Host "文件：$FilePath"
Write-Host "章节：$SectionTitle"
Write-Host "超时阈值：${TimeoutSeconds}秒"
Write-Host ""

# 检查文件是否存在
if (-Not (Test-Path $FilePath)) {
    Write-Host "❌ 文件不存在" -ForegroundColor Red
    exit 1
}

# 检查章节是否存在
$sectionMatch = Select-String -Path $FilePath -Pattern [regex]::Escape($SectionTitle)
if ($sectionMatch) {
    Write-Host "✅ 章节存在（第$($sectionMatch.LineNumber)行）" -ForegroundColor Green
    Write-Host "   说明：子代理超时前已完成写入" -ForegroundColor Green
    exit 0
} else {
    Write-Host "❌ 章节不存在" -ForegroundColor Red
    Write-Host "   说明：需要重新执行增强任务" -ForegroundColor Yellow
    exit 2
}
```

### B5.2 批量恢复脚本

```powershell
# 批量恢复脚本（Windows PowerShell）
# 用法：保存为 batch-recover.ps1，执行 .\batch-recover.ps1

param(
    [string]$TaskList,
    [int]$MaxFilesPerBatch = 2
)

Write-Host "=== 批量恢复脚本 ===" -ForegroundColor Cyan
Write-Host "任务列表：$TaskList"
Write-Host "每批次最大文件数：$MaxFilesPerBatch"
Write-Host ""

$tasks = $TaskList -split ","
$batchCount = [Math]::Ceiling($tasks.Count / $MaxFilesPerBatch)

Write-Host "计算结果：$($tasks.Count)个任务，拆分为$batchCount个批次" -ForegroundColor Yellow
Write-Host ""

for ($i = 0; $i -lt $batchCount; $i++) {
    $start = $i * $MaxFilesPerBatch
    $batch = $tasks[$start..([Math]::Min($start + $MaxFilesPerBatch - 1, $tasks.Count - 1)]]
    Write-Host "批次$($i+1)：$($batch -join ", ")" -ForegroundColor White
}
```
