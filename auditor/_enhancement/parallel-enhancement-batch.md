## 概述

本次知识库增强分三波执行，累计处理39个文件、18项专家理论注入。批次管理是保证大规模知识工程高质量交付的关键机制。

---

## 1. 批次规划原则

| 原则 | 说明 | 阈值 |
|------|------|------|
| **规模控制** | 单批次文件数 | ≤5个（P0）/ ≤6个（P1）/ ≤4个（P2） |
| **超时预防** | 预计执行时间 | ≤3分钟/文件 |
| **并行控制** | 单批次并行子代理数 | ≤3个 |
| **依赖管理** | 批次间依赖 | P0→P1→P2串行 |
| **备份机制** | 主会话备选 | 每个并行批次配置 |

### 阈值依据

- 知识增强单文件平均50~80行内容
- 子代理执行约2~3分钟/文件（含读取+写入+验证）
- 超过3个文件并行 → gateway过载风险显著上升
- 超过4个文件/批 → 超时概率>50%

## 3. 批次执行监控清单

### 执行前检查

```
[ ] 确认增强文件已备份（git add -A && git commit）
[ ] 确认目标章节插入位置（grep "上一章节标题" 文件）
[ ] 确认专家来源信息已收集（来源专家+机构+年份）
[ ] 确认子代理超时阈值已设置（知识增强类→3分钟）
[ ] 确认主会话备选方案已准备（子代理超时→主会话直接输出）
[ ] 确认并行子代理数≤3（gateway过载预防）
```

### 执行中监控

```
[ ] 监控子代理完成状态（等待completion事件）
[ ] 超时检测：超过预期时间2倍 → 检查是否已写入
[ ] 若子代理报告"已存在" → grep验证章节是否真实存在
[ ] 若子代理超时 → 检查文件是否已变更（部分成功）
[ ] 记录每个子代理的完成时间
```

### 执行后验证

```
[ ] grep确认章节标题存在
[ ] 统计行数增长（增强后应>增强前+50行）
[ ] grep "增强来源：" 确认专家来源标注
[ ] 全批次完成后：更新memory-config.md增强记录
```

## 4. 超时恢复流程

当子代理超时时，按以下流程恢复：

```
子代理超时
    ↓
检查文件是否已变更
    ↓
否 → spawn单个文件补做任务
    ↓
是 → grep验证章节是否已存在
    ↓
已存在 → 标记为✅完成
未存在 → 写入增强内容
    ↓
验证（grep行号+行数增长）
```

## 5. 批次间依赖管理

### 依赖图

```
P0批次 ──────────────────┐
  ↓                      │
P1批次 ──────────────────┤──→ 全部完成 → 写memory-config.md
  ↓                      │
P2批次 ──────────────────┘
```

### 依赖管理规则

1. **P0先于P1执行**：P0验证通过后才能启动P1
2. **P1先于P2执行**：P1验证通过后才能启动P2
3. **同批次内并行**：P0/P1/P2批次内各文件独立，可并行执行
4. **恢复任务优先**：若某批次有超时恢复任务，恢复任务优先于后续批次

### 批次启动条件

| 条件 | P0 | P1 | P2 |
|------|----|----|-----|
| 前一批次完成验证 | N/A | ✅ | ✅ |
| 增强文件备份 | ✅ | ✅ | ✅ |
| 专家来源信息收集 | ✅ | ✅ | ✅ |
| 子代理超时阈值设置 | ✅ | ✅ | ✅ |
| 主会话备选方案 | ✅ | ✅ | ✅ |

## 7. 批次规模扩展指南

当知识工程规模扩大时，按以下规则扩展：

| 总文件数 | 建议批次策略 |
|---------|------------|
| 1~5个 | 单批次，并行度≤3 |
| 6~15个 | 3个批次（P0/P1/P2），每批≤5个文件 |
| 16~30个 | 4个批次，每批≤4个文件，增加P3 |
| 30个以上 | 按团队模板分域（金融/法律/内容），域内分批 |

### 跨域依赖管理

```
金融域增强（P0/P1/P2）
    ↓
法律域增强（P0/P1/P2） ───→ 跨域增强协调（主编）
    ↓
内容域增强（P0/P1/P2）
```

## 8. 实际执行数据

| 批次 | 启动时间 | 文件数 | 增强项数 | 并行度 | 超时数 | 完成率 | 总耗时 |
|------|---------|--------|---------|-------|-------|--------|-------|
| P0 | 01:18 | 5 | 7 | 3 | 0 | 100% | ~5分钟 |
| P1 | 01:18 | 6 | 8 | 2 | 0 | 100% | ~5分钟 |
| P2 | 01:36~01:42 | 4 | 4 | 1 | 3→恢复 | 100% | ~15分钟 |

**关键发现**：
- P0+P1并行执行总计5分钟（高效）
- P2因任务被中断，恢复耗时较长（~15分钟）
- 教训：单批次并行度≤3，但仍建议P2用≤2（防止连锁超时）
