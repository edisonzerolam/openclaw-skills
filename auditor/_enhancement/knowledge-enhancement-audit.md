# 增强层A：知识库增强审计模式

> 本增强层基于 ClawTeam 知识库增强实战（39个知识文件，P0/P1/P2共18项专家理论注入）经验沉淀，用于指导 auditor 在审计知识增强类变更时的标准化流程。

---

## A1. 知识库增强的变更分类识别（G1补充）

### A1.1 三类知识变更

知识文件变更按影响类型分为以下三类，auditor 需在 G1 阶段准确识别：

| 变更类型 | 定义 | 审计重点 | 示例 |
|---------|------|---------|------|
| Agent变更 | 人设/角色/行为规范类文件修改 | 一致性、越权风险 | `rules.md`, `agents-config.md` |
| Skill变更 | 技能定义/工具集成/工作流类文件修改 | 接口兼容性、版本冲突 | `SKILL.md`, `tools-config.md` |
| 跨域知识更新 | 专业知识注入（专家理论/数据源/公式） | 来源标注、时效性、交叉引用 | `references/knowledge/*.md` |

### A1.2 本次实战分类案例

本次18项知识增强的分类识别：

```
变更类型混合判断：
- 39个知识文件分布在5个团队模板
  → chatlaw (法律), enterprise-legal (企业法务), 
    a-share-analysis (A股分析), ai-content-creator (AI内容),
    content-distribution (内容分发)
- 每文件包含多个专家理论注入点 → Skill变更+跨域知识更新 混合类型

识别结论：本次变更属于「Skill变更+跨域知识更新」混合类型
         审计时需同时检查 Skill 接口兼容性和知识来源合规性
```

### A1.3 识别规则（ auditor 行动项）

```
G1阶段变更分类检查清单：
[ ] 扫描变更文件路径，判断属于哪类变更
[ ] 跨域知识更新：检查是否在 `references/knowledge/` 目录
[ ] Skill变更：检查是否有 `SKILL.md` 或 `_enhancement/` 目录变更
[ ] 混合类型变更：同时触发 Skill审计线 和 知识审计线
[ ] 记录变更类型到 Phase-G 分类结果
```

## A2. 知识增强的质量门检查（Q扩展）

在标准 Q0-Q6 质量门基础上，知识增强类变更额外增加以下质量门：

### A2.1 Q7 知识文件规模

```
质量门Q7：知识文件规模
- 单文件行数 > 500行 或 文件大小 > 20KB → warn
- 处理方式：拆分为多章节（每章节建议 100-200行）
- 验证：增强后行数应显著增长（每项增强约 50-80行）

本次实战参考：
- stock-analyst.md 增强前约 300行，增强后约 580行（+280行）
- 增长量符合预期：单文件含3项专家理论注入（护城河/估值倍数/宏观配置）
```

### A2.2 Q8 专家来源标注

```
质量门Q8：专家来源标注（强制）
- 每项知识增强必须包含「来源专家+年份」标注
- 格式：[增强来源：专家姓名，年份]
- 无来源标注 → warn 并要求补充
- 可接受来源：学术论文、行业报告、经典著作、专利文档

本次实战标注示例：
```
## 3.2 护城河分析
[增强来源：巴菲特《巴菲特致股东信》，1995]

护城河的五种类型：
1. 无形资产（品牌溢价/专利保护）
2. 转换成本（客户粘性）
3. 网络效应（平台型业务）
4. 成本优势（规模经济/独特资源）
5. 高效规模（细分市场利基）
```

无来源标注的典型错误：
```
## 3.2 护城河分析
护城河是指...（缺失来源标注）  ← warn
```

### A2.3 Q9 公式完整性

```
质量门Q9：公式完整性（LaTeX专项）
- 含 LaTeX 公式的文件需验证公式渲染正确性
- 公式数量 ≥ 3 个 → 需要渲染检查（使用 grep 统计 $$ 或 $ 数量）
- 检查项：
  [ ] 行内公式 $...$ 和 块级公式 $$...$$ 嵌套正确
  [ ] 公式内无未转义的特殊字符（_ 需要写成 \_）
  [ ] 变量名使用英文（避免 $净利润$ → 应改为 $NetIncome$）

本次实战公式检查案例：
```bash
# 检查 stock-analysis.md 中的公式数量
grep -c '\$' stock-analysis.md
# 结果：12个公式，≥3个，需渲染检查
```

### A2.4 Q10 语言一致性

```
质量门Q10：三码混排一致性检查
- 中文正文 + 英文变量名 + LaTeX公式 的混排场景
- 常见问题：
  1. 中英文之间缺少空格 → 影响可读性
  2. 变量名混用中英文 → $净利润$ vs $NetIncome$ 不统一
  3. 全角/半角混用 → （）vs () 不一致

一致性检查命令：
```bash
# 查找可能的中文变量名（错误模式）
grep -n '\$[^\$]*[\u4e00-\u9fa5][^\$]*\$' references/knowledge/*.md
```
```

## A3. 知识增强的Scope边界（S1扩展）

### A3.1 影响范围矩阵

```
S1阶段Scope定义：知识增强的影响范围

影响范围：39个知识文件 × 5个团队模板
┌─────────────────────┬──────────────────┬──────────────────┐
│ 团队模板             │ 知识文件数       │ 增强项数         │
├─────────────────────┼──────────────────┼──────────────────┤
│ a-share-analysis    │ 12个             │ 5项（P0核心）    │
│ chatlaw             │ 8个              │ 4项（P1）        │
│ enterprise-legal    │ 7个              │ 3项（P1）        │
│ ai-content-creator  │ 6个              │ 3项（P2）        │
│ content-distribution│ 6个              │ 3项（P2）        │
└─────────────────────┴──────────────────┴──────────────────┘
总计：39个文件，18项增强，来源专家涵盖：巴菲特/芒格/索罗斯等
```

### A3.2 交叉影响识别

```
交叉影响：同一专家理论可能跨多个文件

案例：「护城河」理论
- 出现位置1：a-share-analysis/stock-analyst.md（价值投资章节）
- 出现位置2：a-share-analysis/valuation-expert.md（估值方法章节）
- 审计要求：两处引用的来源必须一致（均标注巴菲特1995）
- 变更影响：如果修改护城河定义，需同步检查所有交叉引用文件

交叉影响扫描命令：
```bash
# 查找同一关键词在不同文件中的出现
grep -l "护城河" references/knowledge/a-share-analysis/*.md
# 结果：stock-analyst.md valuation-expert.md
```
```

### A3.3 变更路径规范

```
知识增强的标准变更路径：
变更路径前缀：references/knowledge/

正确示例：
- references/knowledge/a-share-analysis/stock-analyst.md
- references/knowledge/chatlaw/legal-expert.md

错误示例：
- skills/stock-analyst/SKILL.md  ← 非知识文件路径
- documents/knowledge/stock-analyst.md  ← 缺少 references 前缀

auditor 行动项：
[ ] 确认变更路径符合 `references/knowledge/` 前缀规范
[ ] 不符合规范 → warn 并要求重新组织文件结构
```

## A4. 知识增强的S2规划特征（S2扩展）

### A4.1 并行化可行性判断

```
S2规划：知识增强任务天然适合并行化

判断依据：
1. 每个知识文件独立无依赖（内容自包含）
2. 专家理论注入点之间无交叉引用修改
3. 文件间隔离良好（不同团队模板）

并行化条件：
- 文件数 ≤ 4个/批次 → 可并行
- 文件数 > 4个/批次 → 拆分为多个批次

本次实战批次划分：
P0批次（5文件7项，并行3个子代理）：
  子代理A：stock-analyst.md + macro-forecasting.md
  子代理B：hk-stock.md + alternative-data.md
  子代理C：valuation-expert.md（单文件，增强项多）

P1批次（6文件8项，分2组并行）：
  组1：legal-expert.md + contract-review.md + compliance-audit.md
  组2：news-analysis.md + regulatory-change.md + litigation-tracker.md

P2批次（4文件4项，串行执行）：
  组1：pipeline-valuation.md
  组2：signal-decay.md
```

### A4.2 批次优先级排序

```
P0优先级（立即生效）：影响核心工作流
P1优先级（下一迭代）：增强现有内容但非核心
P2优先级（后续规划）：可独立存在的专项增强

优先级判断标准：
- P0：文件被主工作流直接引用（如 stock-analyst.md 被交易策略引用）
- P1：文件提供辅助参考（如 legal-expert.md 提供法律咨询参考）
- P2：文件为独立专题（如 pipeline-valuation.md 专题研究）
```

### A4.3 超时风险管理

```
S2超时风险管理：知识增强任务特殊处理

风险识别：
- 知识增强任务为内容写入类，实际执行时间与内容复杂度正相关
- 专家调研任务（5分钟超时）：信息收集+写入，输出不确定性高

超时预防策略：
1. 超时阈值设置：知识增强类 → 3分钟（原5分钟的60%）
2. 任务分解：超过3个文件 → 拆分为≤2文件/子代理
3. 主会话备选：每个并行批次附带主会话备选方案

超时计算：
- 原阈值：5分钟 = 300秒
- 调整阈值：3分钟 = 180秒
- 原因：内容写入类任务写入时间可预估，3分钟足够
- 备选触发：子代理超时 → 主会话直接输出（不等待子代理）
```

## A5. 知识增强的S4验证项（S4扩展）

### A5.1 增强插入位置验证

```
S4验证第一步：确认增强章节插入位置正确

验证命令：
```bash
# 查找目标章节标题是否存在
grep -n "## 3.2 护城河分析" references/knowledge/a-share-analysis/stock-analyst.md
# 结果：123行  ## 3.2 护城河分析

# 验证行号连续性（章节标题后应有内容）
sed -n '123,130p' references/knowledge/a-share-analysis/stock-analyst.md
```

预期结果：
- grep 返回行号（非零退出码表示找到）
- sed 显示章节标题后紧跟内容（非空行）

失败模式：
- grep 返回空 → 章节未插入
- sed 显示空内容 → 插入不完整
```

### A5.2 文件规模增长验证

```
S4验证第二步：确认增强后文件规模符合预期

验证命令：
```bash
# 统计增强前后行数
wc -l references/knowledge/a-share-analysis/stock-analyst.md
# 结果：580行（增强前约300行，增长约280行）

# 批量验证（多个文件）
for f in references/knowledge/a-share-analysis/*.md; do
  echo "$f: $(wc -l < "$f")行"
done
```

增长预期：
- 单项增强：50~80行/项
- 3项增强文件：增长150~240行
- 5项增强文件：增长250~400行

失败模式：
- 增长量为0或负 → 增强未成功
- 增长量过大（>500行）→ 可能存在重复插入
```

### A5.3 专家来源标注验证

```
S4验证第三步：确认专家来源标注存在

验证命令：
```bash
# 查找增强来源标注
grep -n "增强来源：" references/knowledge/a-share-analysis/stock-analyst.md
# 结果：
# 125行 [增强来源：巴菲特《巴菲特致股东信》，1995]
# 187行 [增强来源：芒格《穷查理宝典》，2000]
# 245行 [增强来源：索罗斯《金融炼金术》，1992]

# 批量验证（所有增强文件）
grep -l "增强来源：" references/knowledge/a-share-analysis/*.md | wc -l
# 结果：5个文件全部包含来源标注
```

失败模式：
- grep 返回空 → 缺失来源标注 → warn
- 部分文件缺失 → 未完整覆盖 → warn
```

## A6. 知识增强的S5经验沉淀格式

### A6.1 增强记录标准格式

```
S5阶段完成后的经验沉淀记录：

memory-config.md 或审计报告中记录格式：
```
- [#知识增强] YYYY-MM-DD auditor | {文件数}+{增强项数}+{来源专家数} | ⭐
```

本次实战记录示例：
```
- [#知识增强] 2026-05-17 auditor | 39+18+12 | ⭐⭐⭐
  说明：5个团队模板，39个知识文件，18项专家理论注入，12位来源专家
  批次：P0(5文件7项) / P1(6文件8项) / P2(4文件4项)
  问题：3组子代理超时（已通过主会话备选恢复）
```

### A6.2 增强质量评分

```
增强质量评分标准：

⭐☆☆☆☆：基础完成（来源标注缺失或格式不规范）
⭐⭐☆☆☆：合格（来源标注规范，但规模超标或公式问题）
⭐⭐⭐☆☆：良好（符合Q7-Q10所有质量门）
⭐⭐⭐⭐☆：优秀（交叉影响识别完整，超时管理到位）
⭐⭐⭐⭐⭐：卓越（批次划分合理，验证流程全覆盖，问题自愈成功）

本次实战评分：⭐⭐⭐⭐☆
- 原因：超时恢复成功，但首次批次超时率偏高（60%）
- 改进项：下次批次大小调整为≤3文件/批次
```

### A6.3 经验复用建议

```
S5输出：auditor 执行知识增强任务时的复用建议

复用建议清单：
1. 批次大小：≤4文件/批次（超出→分批）
2. 超时阈值：知识增强类 → 3分钟（调研类 → 2分钟）
3. 主会话备选：必须配置（子代理超时 → 主会话直接输出）
4. 验证流程：grep行号 → wc行数 → grep来源标注，三步确认
5. 来源标注：增强来源格式 [增强来源：专家姓名，年份]（强制）
6. 交叉影响：同理论多文件引用 → 来源必须一致
```

## A7. 快速执行清单（auditor 用）

```
知识增强审计快速执行清单：

□ G1：识别变更类型（Skill变更+跨域知识更新 混合）
□ S1：定义Scope边界（39文件×5团队模板）
□ S2：规划批次（P0/P1/P2三级，≤4文件/批次）
□ Q7：检查文件规模（>20KB→warn）
□ Q8：检查来源标注（缺失→warn）
□ Q9：检查公式渲染（≥3个公式→检查）
□ Q10：检查语言一致性（中文+英文+LaTeX混排）
□ S4：执行验证（grep行号 → wc行数 → grep来源）
□ S5：记录经验（增强记录格式→memory-config.md）

执行优先级：G1 → S1 → S2 → (Q7-Q10) → S4 → S5
```
