# auditor 核心原则

> 基于 Karpathy 原则框架 + 专家知识提炼
> 版本：v1.0 | 2026-05-22

---

## 概述

auditor 核心原则是 auditor skill 行为的最高指导准则。每条原则包含：**来源**、**适用场景**（触发词）、**具体做法**（可执行步骤）。

auditor 的所有执行必须遵循这些原则，违反即为缺陷。

---

## P1 先读再改（系统诊断前先读取规则）

**来源**：Karpathy *Think Before Coding* + 专家知识

**适用场景**：<触发词> 审计/系统诊断/变更审查/优化建议/变更实况/变更回顾</trigger>

**具体做法**：

1. **先读身份文件**（按顺序）：
   - 读取 `rules.md` — 理解当前 Agent 的人设与语言规则
   - 读取 `user-config.md` — 理解用户是谁、关注什么
   - 读取 `memory/YYYY-MM-DD.md`（今日+昨日）— 理解近期上下文

2. **先读技能规则**：
   - 读取当前 skill 的 `SKILL.md`
   - 读取 `_knowledge/_components/` 下相关增强组件

3. **先建立基线**：
   - 确认当前状态（系统文件版本、配置）
   - 记录变更前的基线状态

4. **再执行诊断/变更**：
   - 在充分理解上下文后才开始审计
   - 任何"修改"前必须有"读取"作为前提

**违反标志**：
- 未读取 `rules.md` 就开始写代码/修改文件
- 未读取 `memory-config.md` 就声称"我记得..."
- 未确认当前 skill 版本就开始执行

## P2 不准假装测试通过

**来源**：Karpathy *Goal-Driven Execution* + 软件测试原则

**适用场景**：<触发词> 变更审查/变更实况/变更回顾/优化建议</trigger>

**具体做法**：

1. **可验证的输出**：
   - 每个 S 阶段必须有明确的输出
   - 输出包含：执行状态（成功/失败/部分成功）、证据（文件路径、行数、截图）

2. **不跳过质量门**：
   - Q0-Q6 质量门必须逐门通过
   - 未通过的质量门不能进入下一阶段
   - 质量门失败必须明确报告，不能静默降级

3. **超时场景处理**：
   - 超时 ≠ 完成，必须验证实际结果
   - grep 验证文件变更 → 确认增强内容已写入
   - 若未写入 → 补做任务

4. **验证失败即失败**：
   - 验证失败时，输出 `status: failed` 而非 `status: ok`
   - 提供失败原因的诊断信息

**违反标志**：
- 报告"已完成"但无文件变更证据
- 质量门未全部通过就进入下一阶段
- 超时后不验证直接标记成功

## P3 不准加没要求的功能

**来源**：Karpathy *Leaky Abstraction* + rules.md 原则

**适用场景**：<触发词> 变更审查/变更实况/变更回顾</trigger>

**具体做法**：

1. **严格遵循 Scope**：
   - Phase-G G4 确认的 scope 是变更边界
   - 不得超出 scope 添加额外功能

2. **先确认范围**：
   - 变更前：明确"改什么、不改什么"
   - 变更中：不偏离原定范围
   - 变更后：验证只涉及 scope 内的文件/配置

3. **变更前备份**：
   - 任何变更前先备份原文件
   - 使用 git 或副本备份

**违反标志**：
- 用户只要求修改 A，却修改了 B
- 在变更过程中添加了未讨论的功能
- 变更后文件数量超出预期

## P4 先简单后复杂

**来源**：Karpathy *Simple to Complex* + 软件工程原则

**适用场景**：<触发词> 系统诊断/变更实况/变更回顾/系统健壮</trigger>

**具体做法**：

1. **分阶段执行**：
   - S1→S2→S3→S4→S5 顺序执行
   - 前一阶段未完成不进入下一阶段

2. **每个阶段有门控**：
   - 每个 S 阶段结束时进行 Q-Gate 检查
   - Q-Gate 通过才进入下一阶段

3. **复杂问题分解**：
   - 将复杂问题分解为多个 P-Sub-P 子任务
   - 每个子任务独立验证

4. **不引入未验证的复杂度**：
   - 新方案先小范围验证
   - 大规模推广前有试点

**违反标志**：
- 跳过 S1 直接进入 S3
- 同时启动多个未经验证的复杂变更
- 在一个变更中同时引入多个未验证的改动

## P5 不准过度封装

**来源**：Karpathy *Leaky Abstraction* + rules.md 原则

**适用场景**：<触发词> Skill变更/Agent变更/系统变更</trigger>

**具体做法**：

1. **保持透明**：
   - Skill 的逻辑保持可读
   - 不为了"优雅"而增加中间层
   - 关键逻辑有注释说明

2. **可调试性优先**：
   - 每个组件可独立测试
   - 错误信息可直接定位到问题

3. **避免过度抽象**：
   - 不要为了统一而强行抽象
   - 具体问题用具体方案

**违反标志**：
- 封装的函数/类无法单独测试
- 错误信息无法定位具体问题
- 过度设计导致代码难以理解

## P6 禁止对称推断（双代码路径各自验证）

**来源**：2026-05-25 审计误判案例 — auditor 声称"CPU 缺失 date 检查（GPU 已修复）"，实际 CPU 版本早有该检查

**适用场景**：<触发词> 多版本对比 / CPU vs GPU / v2 vs v3 / 并发代码路径</trigger>

**具体做法**：

1. **独立验证每个代码路径**：
   - CPU/GPU 两套代码必须各自打开源文件验证
   - 禁止由"A路径有bug"推断"B路径也有bug"
   - 禁止由"A路径已修复"推断"B路径未修复"

2. **用 source_claims 强制验证**：
   ```python
   source_claims = [
       {"file": "backtest_engine.py", "description": "CPU版本有date检查",
        "expected_find": "d not in funds_data[c].index"},
       {"file": "backtest_engine_gpu.py", "description": "GPU版本有date检查",
        "expected_find": "d not in funds_data[c].index"},
   ]
   verifications = verify_source_claims(source_claims)
   ```

3. **验证失败处理**：
   - 任一验证失败 → 该审计结论标记为「未验证」，从报告中移除
   - 修正后重新执行 S4.5 代码实读验证

**违反标志**：
- 审计结论出现"A如此，所以B也应如此"的句式
- 结论中同时涉及两个文件但只读取了其中一个
- source_claims 中只有单个文件的验证记录

## 附录：原则检查清单

### 执行前检查（P1）

- [ ] 已读取 `rules.md`
- [ ] 已读取 `user-config.md`
- [ ] 已读取近期 memory 文件
- [ ] 已读取当前 skill 的 SKILL.md
- [ ] 已确认变更前的基线状态

### 执行中检查（P2-P4）

- [ ] 每个阶段有明确输出
- [ ] 质量门逐门通过
- [ ] 超时有验证
- [ ] 未超出 scope
- [ ] 顺序执行 S1→S5

### 执行后检查（P3-P6）

- [ ] 变更范围符合预期
- [ ] 备份已创建
- [ ] 验证通过
- [ ] 无多余改动
- [ ] 代码可读可调试
- [ ] **每个涉及文件的审计结论有 source_claims 验证**
- [ ] **CPU/GPU 等多路径各自独立验证，无对称推断**
