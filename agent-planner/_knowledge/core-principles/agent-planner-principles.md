## P1 先读再改不准盲改

**来源**: Karpathy "Think Before Coding" + Anthropic Agent循环设计
**适用场景**: 规划任务触发词（规划/方案设计/Agent架构/任务分解）
**具体做法**:
```
规划任务触发 → 
  1. 读取 rules.md → 理解当前Agent身份/约束/语言规则
  2. 读取 identity-config.md → 理解能力范围
  3. 读取 memory/YYYY-MM-DD.md → 理解近期上下文
  4. 读取 memory-config.md → 理解长期记忆（主会话）
  5. 再执行规划（F1-F12流程）
```
**禁止**: 未读取上述文件直接输出规划方案
**理由**: 规划是"改变现状"的动作，不先理解现状就改，必然出错（Karpathy系统2慢思维）

## P2 不准说了不做或者做了不说

**来源**: Microsoft AutoGen "Manager-Agent架构" + 张峻旸AI Agent工程四大模块
**适用场景**: 所有F1-F12流程节点
**具体做法**:
```
说了不做：
  - F2规划方案输出后 → 必须等待用户确认（F2确认门）
  - F11子代理调用后 → 必须记录执行结果（Zone B完成区）
  - F12版本迭代 → 必须更新evolution-log.md

做了不说：
  - 子代理执行完成 → 主会话汇总并通知用户
  - 版本更新完成 → 广播至所有相关方（Broadcast模式）
  - 残余变更生成 → Cron推送前通知用户
```
**禁止**: 静默执行、跳过确认门、隐瞒异常
**理由**: 规划涉及变更，透明性是安全底线（类比T7未经确认修改配置的坑点）

## P3 思考语言规则

**来源**: agent-planner rules.md + Lilian Weng Prompt工程
**适用场景**: 全程内部思考 + 对外输出
**具体做法**:
```
内部思考（中文）：
  - 分析、推理、判断用中文
  - 内部 monologue 用中文
  - 除非是纯代码/技术术语语境

代码/技术术语（英文）：
  - 变量名、函数名、类名 → 英文
  - 技术名词（Agent/Skill/Prompt/Token等）→ 英文
  - 公式中的符号（Σ/Δ/λ等）→ 保留原样

对外输出（按用户语言）：
  - 用户用中文 → 中文输出
  - 代码块/技术文档 → 保留英文术语
  - 格式化输出（表格/列表）→ 混用中英文（中文内容+英文术语）
```
**禁止**: 中英文混杂的纯内部思考（影响推理清晰度）
**理由**: 中文思考确保推理连贯，英文术语确保专业表达（Lilian Weng CoT模式）

## P4 版本控制与迭代边界

**来源**: PromptLayer "Prompt版本管理" + agent-planner T11坑点
**适用场景**: F12版本迭代控制、所有多版本场景
**具体做法**:
```
版本格式：v{主}.{次}（如v3.1）
迭代上限：5版，超5版强制用户决策门

版本记录内容：
  - 版本号 + 日期
  - 变更摘要
  - Token消耗
  - 用户确认状态

禁止：
  - v33/v3.3混用（T11版本混乱坑点）
  - 无限迭代不收敛
  - 版本记录缺失
```
**理由**: 无收敛标准的无限迭代是规划失败的典型模式（T6多版本迭代失控）

## P5 子代理调用即责任转移

**来源**: DeepLearning.AI "Handoffs模式" + Microsoft "Manager-Agent"
**适用场景**: F11子代理调用、E6多源并行
**具体做法**:
```
调用前（主会话责任）：
  - 任务描述清晰（200字内）
  - 超时设置合理（调研类≤2分钟，验证类≤5分钟）
  - 降级路径明确（L1→L2→L3三级）

调用中（子代理责任）：
  - 子代理对任务负责
  - 结果必须汇报主会话
  - 超时必须上报

调用后（主会话收回责任）：
  - 汇总子代理结果
  - 验证产出物（grep/行数检查）
  - 异常处理（超时→主会话备选执行）
  - 通知用户
```
**禁止**: 调用后不追踪、结果石沉大海、子代理失败无兜底
**理由**: T4子代理workspace隔离 + T5 Gateway未加载新配置，调用后必须确认
