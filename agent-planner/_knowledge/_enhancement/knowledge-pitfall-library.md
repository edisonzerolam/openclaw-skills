## 1. 超时相关坑点

### 坑点1.1：调研类任务全部超时

**现象**：P0调研组5分钟内全部超时，未产出任何文件。

**根因分析**：
- 调研+写入双任务耗时长
- 子代理5分钟超时对调研任务不够
- 调研任务输出不确定性高（需要搜索+理解+写作）

**解决方案**：
- 调研类任务超时阈值从5分钟→2分钟
- 主会话备选：子代理超时→主会话直接输出
- 任务拆分：调研和写入分离，先调研再写入

**代码示例**（设置超时阈值）：
```python
# 调研类任务：2分钟超时
spawn(timeoutSeconds=120, task="调研XX专家理论...")

# 知识增强类任务：3分钟超时
spawn(timeoutSeconds=180, task="增强XX文件...")
```

**教训**：并行批次中超过3个文件 → 必有超时，控制在≤3个/批。

---

### 坑点1.2：子代理超时不等于未执行

**现象**：子代理报告"超时"，但文件可能已经写入成功。

**根因**：子代理内部超时（Agent Result timeout）先于操作完成触发，但操作本身已执行。

**解决方案**：
- 超时后先grep目标章节标题，确认是否已存在
- 行数统计验证：增强后行数是否>增强前+50行

**验证流程**：
```bash
# 超时后立即执行验证
Select-String "第X节标题" 文件.md
# 如果返回行号 → 内容已写入，标记✅完成
# 如果无返回 → 需要补做
```

**教训**：超时是执行效率问题，不是执行结果问题。

---

### 坑点1.3：批量超载导致连锁超时

**现象**：1个批次同时spawn 4+个子代理，全部超时。

**根因**：Gateway并发处理能力有限，超过3个并行子代理会导致调度过载。

**解决方案**：
- 单批次并行子代理数 ≤ 3个
- 超载后立即拆分为更小批次
- 使用串行恢复：逐个补做而非并行补做

**本次教训**：
- P2补做时采用单个文件逐个补做
- 从4个并行→1个串行，最终100%完成

## 2. 插入位置错误坑点

### 坑点2.1：章节编号冲突

**现象**：计划插入"第3.6节"但文件已有3.6节（之前的增强已插入）。

**根因**：多批次增强后，章节编号与原计划不符。

**解决方案**：
- 明确插入位置用"在现有第X节之后新增"而非"第X节编号"
- 先grep确认目标章节是否存在
- 插入后grep验证行号

**验证命令**：
```bash
# 确认第3.5节是否存在（插入位置确认）
Select-String "## 3\.5" hk-stock-analysis.md

# 确认插入后第3.6节存在（验证）
Select-String "## 3\.6" hk-stock-analysis.md
```

**教训**：章节编号是动态的，插入位置描述用"在XX之后"而非编号。

---

### 坑点2.2：插入位置计算错误

**现象**：grep显示插入位置，但实际写入位置偏移。

**根因**：文件有多处匹配的"## 第X节"，grep选择错误匹配行。

**解决方案**：
- 使用精确匹配：`Select-String "^## X\.Y " 文件`（行首+精确编号）
- 验证插入后内容上下文是否正确

---

## 3. 内容覆盖坑点

### 坑点3.1："已存在"不等于未执行

**现象**：子代理报告"章节已存在"，但实际增强内容是在上一次P1批次中已成功写入。

**根因**：子代理检测逻辑：若存在同名章节→退出。但上一次增强已成功写入，子代理误判为"无需执行"。

**解决方案**：
- grep + 行数统计双重验证
- 先检查增强后行数是否>增强前+50行
- 若行数正常增长 → 已执行，标记✅

**本次教训**：
- hk-stock-analysis.md 第3.6节：子代理报告"已存在"，grep确认第188行存在
- 说明P1批次已成功插入，子代理只是重复检测后退出

---

### 坑点3.2：覆盖而非追加

**现象**：增强内容写入后，原有内容被覆盖。

**根因**：编辑操作使用"替换"模式而非"追加"模式。

**预防**：只在确认增强章节完全不存在时才执行写入操作。

## 4. 语言混排坑点

### 坑点4.1：LaTeX公式内含中文变量名

**现象**：`$净利润$` 在某些markdown渲染器中无法正确显示。

**根因**：LaTeX数学模式默认不支持CJK字符。

**解决方案**：
- 建立变量名规范：英文/驼峰/下划线
- 变量名对照表：

| 中文含义 | 正确写法 | 错误写法 |
|---------|---------|---------|
| 净利润 | `$NetIncome$` | `$净利润$` |
| 毛利率 | `$GrossMargin$` | `$毛利率$` |
| 市盈率 | `$PE$` | `$市盈率$` |
| 净资产收益率 | `$ROE$` | `$净资产收益率$` |

**检查命令**：
```bash
# 查找可能的格式错误
Select-String '\$[^$]*[\u4e00-\u9fff]' 文件.md
# 匹配公式内含中文的情况
```

---

### 坑点4.2：markdown表格内含复杂LaTeX

**现象**：markdown表格内嵌入LaTeX公式导致渲染错乱。

**解决方案**：
- 复杂公式使用块级LaTeX（`$$...$$`），独立一行
- 表格内只放数值/文本，不放公式
- 优先使用纯markdown表格

---

### 坑点4.3：中英混排无规律

**现象**：同一个文件内，有时中文句末加标点，有时英文标点。

**解决方案**：
- 中文正文使用中文标点（，。：；）
- 英文变量名/公式使用英文标点（.,:;）
- 建立写作规范（本文档后续补充）

## 5. 版本兼容坑点

### 坑点5.1：UTF-8 BOM处理不一致

**现象**：不同skill对UTF-8 BOM处理不同，导致文件乱码或格式破坏。

**解决方案**：
- 使用 `qclaw-text-file` skill 处理所有文本文件写入
- 脚本自动处理BOM/编码/换行符

**检查命令**：
```bash
# 检查文件编码
file 文件.md
# 或在PowerShell中
Get-Content 文件.md -Encoding Byte -TotalCount 3
# EF BB BF = UTF-8 with BOM
```

---

### 坑点5.2：Windows vs Unix换行符

**现象**：文件在Windows用记事本打开正常，但在Unix系统（WSL）打开后格式错乱。

**根因**：CRLF（Windows）vs LF（Unix）换行符差异。

**解决方案**：
- 统一使用qclaw-text-file skill处理换行符
- 换行符规范：`.md` 文件使用LF（Unix风格）

---

### 坑点5.3：路径分隔符差异

**现象**：`C:\Users\...` 在WSL中使用时路径错误。

**解决方案**：
- Windows路径保持原样
- 跨平台操作时使用路径规范化函数

## 6. 增强衰减坑点

### 坑点6.1：增强来源格式不统一

**现象**：不同文件的"增强来源"格式不一致，有的写"来源："，有的写"*增强来源："，有的没有。

**解决方案**：
- 建立统一模板：`*{来源专家（机构）}*{主题}*{年份}*`
- 示例：`*增强来源：黄咏衫（高盛香港）2023~2024年港股18A估值方法*`

---

### 坑点6.2：跨文件重复内容

**现象**：同一专家理论在多个文件中重复出现，格式和数据不一致。

**解决方案**：
- 专家池统一管理：每个专家理论只在一处主文件定义
- 其他文件引用：`*参考：XX文件*`

---

## 7. 批次管理坑点

### 坑点7.1：批次间依赖未管理

**现象**：P2在P1验证完成前启动，导致失败后需要重新执行。

**解决方案**：
- 批次门控：P0验证完成→启动P1；P1验证完成→启动P2
- 使用串行门禁而非并行启动

**教训**：批次启动前必须确认前置批次验证通过。

---

### 坑点7.2：批次内并行度过高

**现象**：1个批次4个文件并行spawn，全部超时。

**解决方案**：
- 单批次并行度 ≤ 3
- 超出自动拆分：4个文件→拆成2+2两个批次

---

## 8. 验证坑点

### 坑点8.1：grep验证不完整

**现象**：grep找到章节标题，但章节内容为空（仅标题行）。

**解决方案**：
- 行数验证：增强后行数 > 增强前 + 50行
- 内容验证：grep确认章节内含有效内容（公式/表格/案例）

**完整验证流程**：
```bash
# 1. 章节标题存在
Select-String "## X.Y 章节标题" 文件.md
# 2. 行数增长
(Get-Content 文件.md).Count
# 3. 增强来源存在
Select-String "增强来源：" 文件.md
# 4. 公式存在（如有）
Select-String '\$' 文件.md
```
